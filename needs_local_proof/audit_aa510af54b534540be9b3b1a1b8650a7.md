## Attacker Path
**preconditions:**
- A token exists whose `quoteTokenAddress` is an ERC20 (e.g. USDT) and `nativeToQuoteSwapEnabled == true` (i.e. `QuoteTokenConfiguration.nativeToQuoteSwapType != SWAP_DISABLED`)
- Token is still on the bonding curve (status `ACTIVE`)

**attacker-controlled inputs:**
- `params.inputToken = address(0)`, `params.outputToken = <token>`, `params.inputAmount = X BNB`, `msg.value = X BNB`
- `params.minOutputAmount` set low (or zero) to allow slippage

**call sequence:**
1. Attacker calls `swapExactInput({inputToken: address(0), outputToken: token, inputAmount: X, minOutputAmount: 0, ...})` with `msg.value = X BNB`
2. Portal delegates to `PORTAL_TRADE_V2`
3. `PORTAL_TRADE_V2` detects `inputToken == address(0)` and `quoteToken != address(0)` → triggers nativeToQuoteSwap path
4. Calls `MULTI_DEX_ROUTER.swapExactTokensForTokens` (or `exactInputSingle`) with `X BNB`, receives `Y USDT` where `Y < X_equivalent` due to DEX slippage/fees
5. **Hypothesis:** `PORTAL_TRADE_V2` credits the reserve with `X_equivalent` (derived from `msg.value`) rather than the actual `Y USDT` received
6. Bonding curve state: `reserve += X_equivalent`, but Portal only holds `Y USDT`
7. Attacker (or a second address) immediately sells tokens against the inflated reserve, receiving more USDT than the curve should allow, draining the gap `X_equivalent - Y`

## Why Existing Checks Fail
The `minOutputAmount` field in `ExactInputParams` guards the final bonding-curve token output to the buyer, not the intermediate USDT amount credited to the reserve. If the implementation passes `msg.value`-derived BNB-equivalent (rather than the router's returned `amountOut`) into `estimateSupply` / the reserve update, the slippage loss is silently absorbed into an overstated reserve. The `MULTI_DEX_ROUTER` interface returns actual `amountOut` on every swap path, so the correct value is available — the question is whether the `PORTAL_TRADE_V2` facet uses it. [9](#0-8) 

## Rejection Checks
**expected behavior checked:** Normal slippage chosen by the user is expected behavior — but this claim is that Portal's *internal accounting* uses the wrong amount, not that the user receives fewer tokens. That is not expected behavior.

**prior report checked:** No prior report found in the indexed source.

**README/NatSpec checked:** The NatSpec for `swapExactInput` describes the nativeToQuoteSwap path but does not document any intentional use of `msg.value` for reserve accounting rather than actual USDT received. [10](#0-9) 

**unsupported assumption checked:** The critical assumption — that `PORTAL_TRADE_V2` credits `msg.value`-equivalent rather than actual router output — **cannot be verified from the indexed source**. The `PORTAL_TRADE_V2` facet implementation is not present in the indexed files (`src/Portal.sol`, `src/PortalBase.sol`, `src/PortalCommon.sol`). This is the single blocking uncertainty.

## Local Proof Required
**test type:** BSC fork test (Foundry `--fork-url`)

**test file to add:** `test/NativeToQuoteSwapReserveAccounting.t.sol`

**test setup:**
1. Fork BSC at a recent block
2. Identify a live USDT-quote token with `nativeToQuoteSwapEnabled == true` (query `getTokenV8` on the proxy `0xe2ce6ab80874fa9fa2aae65d277dd6b8e65c9de0`)
3. Record `state.reserve` and `IERC20(USDT).balanceOf(portal)` before the call
4. Call `swapExactInput({inputToken: address(0), outputToken: token, inputAmount: 1 ether, minOutputAmount: 0})` with `msg.value = 1 ether`
5. Record `state.reserve` and `IERC20(USDT).balanceOf(portal)` after the call
6. Compute `deltaReserve = state.reserve_after - state.reserve_before`
7. Compute `deltaBalance = USDT.balanceOf(portal)_after - USDT.balanceOf(portal)_before`

**expected assertion (bug present):**
```
assertGt(deltaReserve, deltaBalance, "reserve overstated vs actual USDT received");
```

**failure condition (bug absent):**
```
assertEq(deltaReserve, deltaBalance, "reserve matches actual USDT received");
```

If `deltaReserve > deltaBalance`, the reserve is inflated by the slippage amount and a subsequent sell will extract more USDT than was deposited, confirming the fund extraction path. If they are equal, the implementation correctly uses the router's `amountOut` and the hypothesis is false.
