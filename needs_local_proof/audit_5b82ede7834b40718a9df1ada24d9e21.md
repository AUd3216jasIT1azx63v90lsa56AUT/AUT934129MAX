## Attacker Path

**preconditions:**
- A victim has signed a valid EIP-2612 permit for the quote token, authorizing Portal as spender for at least `inputAmount`, with a future deadline and the victim's current nonce.
- The attacker observes this permit signature in the mempool (or from a prior on-chain transaction that did not consume the nonce).
- The victim's permit nonce has not yet been incremented (permit not yet consumed).

**attacker-controlled inputs:**
- `params.inputToken` = quote token address (same as victim's permit token)
- `params.outputToken` = any valid bonding-curve token
- `params.inputAmount` = amount covered by victim's permit
- `params.permitData` = victim's raw permit bytes (`owner=victim, spender=portal, value, deadline, v, r, s`)
- `msg.sender` = attacker

**call sequence:**
1. Attacker calls `Portal.swapExactInput(params)` with victim's `permitData` and attacker as `msg.sender`.
2. Portal delegates to `PORTAL_TRADE_V2` implementation via `_delegateToImpl`.
3. Implementation decodes `permitData` and calls `quoteToken.permit(victim, portal, amount, deadline, v, r, s)` — succeeds because signature is valid.
4. Implementation calls `quoteToken.transferFrom(victim, portal, amount)` — succeeds because allowance was just set.
5. Portal mints/transfers output tokens to `msg.sender` (attacker).

**Result:** Victim's quote token balance is consumed; attacker receives the purchased tokens.

## Why Existing Checks Fail

The `swapExactInput` entry point in `Portal.sol` applies only the `onlyIfBitFlagsSet(CB_BIT_MASK_GLOBAL_SWITCH)` guard before delegating — no check that `msg.sender` matches the permit owner. [3](#0-2) 

The EIP-2612 `permit()` call itself does not enforce that `msg.sender == owner`; it only verifies the ECDSA signature over `(owner, spender, value, nonce, deadline)`. If the implementation decodes `owner` from `permitData` and uses that address as the `transferFrom` source (rather than forcing `owner = msg.sender`), the permit succeeds and funds are pulled from the victim.

The OZ `IERC20Permit` documentation included in this repo explicitly warns that the safe pattern requires `msg.sender` as the owner: [4](#0-3) 

**Critical gap:** The actual permit-handling logic lives in the `PORTAL_TRADE_V2` facet (set as an immutable in `PortalBase`). This facet is not present in the indexed source files (`src/Portal.sol`, `src/PortalBase.sol`, `src/PortalCommon.sol`). Whether the implementation enforces `owner == msg.sender` before calling `transferFrom` cannot be confirmed from the indexed source alone. [5](#0-4) 

## Rejection Checks

**expected behavior checked:** EIP-2612 permits are caller-agnostic by design; using them without binding `owner` to `msg.sender` is a known vulnerability class, not expected safe behavior.

**prior report checked:** Cannot confirm from indexed source; no NatSpec or README in the indexed files documents this as intentional or previously reported.

**README/NatSpec checked:** `ExactInputParams.permitData` is described only as "Optional permit data for the input token" with no note that `owner` must equal `msg.sender`.

**unsupported assumption checked:** The attack does not require oracle manipulation, admin compromise, or malicious token behavior — only a valid in-flight or replayable permit signature and the implementation using `owner` (from bytes) rather than `msg.sender` as the fund source.

## Local Proof Required

**test type:** Foundry fork test against BSC mainnet (block ~108382650)

**test file to add:** `test/PermitReplaySwap.t.sol`

**test setup:**
1. Fork BSC at the live block; use proxy `0xe2ce6ab80874fa9fa2aae65d277dd6b8e65c9de0`.
2. Identify a live quote token with EIP-2612 support (e.g., a configured ERC-20 quote token from `_quoteTokenConfigurations`).
3. Fund `victim` with quote tokens; generate a valid permit signature (`owner=victim, spender=portal, value=X, deadline=future`).
4. Do **not** submit the permit from victim's address.
5. From `attacker` (a separate address with zero quote token balance), call `portal.swapExactInput(ExactInputParams{inputToken: quoteToken, outputToken: someToken, inputAmount: X, minOutputAmount: 0, permitData: victimPermitBytes, extensionData: ""})`.

**expected assertion:**
- `quoteToken.balanceOf(victim)` decreases by `X` (or close to it after fees).
- `outputToken.balanceOf(attacker)` increases.
- `quoteToken.balanceOf(attacker)` remains zero (attacker paid nothing).

**failure condition:** If the implementation reverts (e.g., because it calls `transferFrom(msg.sender, ...)` and attacker has no balance, or because it validates `owner == msg.sender` before the permit call), the attack does not work and this should be REJECT.
