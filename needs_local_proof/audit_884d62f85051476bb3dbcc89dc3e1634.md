## Attacker Path

**preconditions:**
- A token exists in `TokenStatus.Staged` state (set by `stageNewTokenV5`, which is restricted to `SALE_FORGE`). The attacker does not need to stage the token themselves; they only need a Staged token to exist on-chain. Since `SALE_FORGE` is a deployed contract that creates staged tokens in the normal protocol flow, this precondition is realistic.
- `bitFlags` has `CB_BIT_MASK_GLOBAL_SWITCH` set (normal operating state).

**attacker-controlled inputs:**
- `params.inputToken` = address(0) (native BNB, for a buy)
- `params.outputToken` = address of a Staged token
- `params.inputAmount` = any nonzero amount
- `params.minOutputAmount` = 0

**call sequence:**
1. Identify a token address with `state.status == TokenStatus.Staged` (e.g., via `getTokenV8Safe`).
2. Call `swapExactInput({inputToken: address(0), outputToken: stagedToken, inputAmount: X, minOutputAmount: 0, permitData: ""})` with `msg.value = X`.
3. `_getSwapImplementation` reads `state.status == Staged`, falls into the `else` branch, returns `PORTAL_TRADE_V2`.
4. `_delegateToImpl(PORTAL_TRADE_V2)` executes the trade logic in the context of Portal's storage.

The question is what `PORTAL_TRADE_V2` does next. The comment says "Let PortalTradeV2 handle the error," implying the developer expects `PORTAL_TRADE_V2` to revert. If `PORTAL_TRADE_V2` does check `state.status != Tradable` and reverts with `TokenNotTradable`, there is no vulnerability. If it does not check status and proceeds to bonding-curve math with `reserve = 0` and `circulatingSupply = 0`, the outcome depends on the curve math — a zero-reserve bonding curve could revert (division by zero) or, in a degenerate case, return a large token amount for a small input.

## Why Existing Checks Fail

The only status check in `_getSwapImplementation` is a positive match for `DEX` and `Tradable`. There is no explicit `revert` for `Staged`, `Invalid`, `Killed`, or `InDuel` at the dispatcher level. [3](#0-2) 

The `TokenNotTradable` error is defined in the interface and exists for use by implementations, but it is not enforced in `Portal.sol` itself before the delegatecall. [4](#0-3) 

The `stageNewTokenV5` gate (`OnlySaleForge`) prevents arbitrary token staging but does not prevent an attacker from targeting an already-staged token. [5](#0-4) 

## Rejection Checks

**expected behavior checked:** The comment "Let PortalTradeV2 handle the error" is the developer's stated intent — `PORTAL_TRADE_V2` is expected to reject non-Tradable tokens. If it does, this is expected behavior and not a bug. This is the primary uncertainty.

**prior report checked:** Not determinable from available source.

**README/NatSpec checked:** No NatSpec on `_getSwapImplementation` documents the Staged-status fallthrough as intentional or safe.

**unsupported assumption checked:** The claim that `PORTAL_TRADE_V2` does not check status is an assumption. The `PORTAL_TRADE_V2` source is not present in this repository (it is a separate delegatecall facet). The actual behavior — whether it checks `state.status` before executing bonding-curve math — cannot be confirmed from the available indexed source alone.
