## Attacker Path

**preconditions:**
- Attacker controls an address that is `_tokenBeneficiaries[token]` for some token whose `lpLocks[token]` is non-zero (i.e., a V4/PCS Infinity CL LP lock exists with accumulated fees)
- Attacker deploys a receiver contract that re-enters `Portal.claim(token)` from its `receive()` fallback
- Attacker sets that receiver contract as the beneficiary (or is already set as one)

**attacker-controlled inputs:**
- `token` address passed to `claim(token)`
- Receiver contract with malicious `receive()` that calls `Portal.claim(token)` again

**call sequence:**
1. Attacker calls `Portal.claim(token)`
2. Portal `_delegateToImpl(PORTAL_ROLLER)` — no reentrancy guard applied before or after
3. PORTAL_ROLLER (delegatecall context) reads `lpLocks[token]`, calls external locker `collectFees` → locker sends ETH to Portal via `Portal.receive()` (plain payable, no logic)
4. PORTAL_ROLLER sends collected ETH to `_tokenBeneficiaries[token]` (attacker's receiver contract)
5. Attacker's `receive()` re-enters `Portal.claim(token)`
6. If PORTAL_ROLLER has not yet cleared `lpLocks[token]` state before the ETH transfer in step 4, the re-entrant call repeats steps 3–4, draining additional BNB

## Why Existing Checks Fail

A grep across all `src/**/*.sol` files for `nonReentrant`, `ReentrancyGuard`, `_reentrancy`, `_locked`, and `_status` returns **zero matches**. [1](#0-0) 

`Portal.claim` applies only `onlyIfBitFlagsSet(CB_BIT_MASK_GLOBAL_SWITCH)` — a liveness check, not a reentrancy guard. [1](#0-0) 

`_delegateToImpl` is a raw assembly delegatecall with no mutex. [3](#0-2) 

`receive()` is `receive() external payable {}` — no logic, no guard, no callback restriction. [2](#0-1) 

**Critical gap**: The PORTAL_ROLLER implementation contract is not present in this repository (only `src/Portal.sol`, `src/PortalBase.sol`, `src/PortalCommon.sol`, and interfaces are indexed). PORTAL_ROLLER is an immutable address set at construction time pointing to a separately deployed contract. [6](#0-5)  Whether PORTAL_ROLLER's `claim` implementation clears `lpLocks[token]` state **before** or **after** transferring ETH to the beneficiary is the decisive question that cannot be answered from the available source. If state is cleared after the ETH transfer (checks-effects-interactions violation), the reentrancy is exploitable.

**Note on the question's described path**: The question states "Portal.receive() → claim again." This is slightly imprecise — `Portal.receive()` is called when the locker sends ETH *to* Portal (step 3 above) and contains no logic. The actual reentrancy trigger is when Portal/PORTAL_ROLLER sends ETH *to the beneficiary* (step 4), causing the attacker's `receive()` to fire. The underlying concern is valid regardless of this description error.

## Rejection Checks

**expected behavior checked:** Beneficiary claiming LP fees is expected behavior; the question is whether the implementation enforces checks-effects-interactions or a reentrancy guard — cannot confirm from available source. [7](#0-6) 

**prior report checked:** No prior report or known-issues file is present in the repository.

**README/NatSpec checked:** No NatSpec on `claim` in Portal.sol beyond the interface delegation comment; no documented reentrancy protection. [1](#0-0) 

**unsupported assumption checked:** The attack requires PORTAL_ROLLER to lack a reentrancy guard AND to transfer ETH to the beneficiary before zeroing `lpLocks`. Both conditions are unverifiable without PORTAL_ROLLER source. The theory is not self-evidently false — no guard exists anywhere in the in-scope files.

**test setup:**

**expected assertion:**
- If vulnerable: `MaliciousReceiver` receives `2 × legitimateFeeAmount` and `Portal.balance` decreases by `2 × legitimateFeeAmount`

**failure condition (proves safe):** Second `claim` call reverts (reentrancy guard present in PORTAL_ROLLER) or returns `(0, 0)` because `lpLocks` was already zeroed before the ETH transfer.
