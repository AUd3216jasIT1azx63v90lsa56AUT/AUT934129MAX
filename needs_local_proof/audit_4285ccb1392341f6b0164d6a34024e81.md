## Attacker Path
**preconditions:**
- A token exists with `quoteTokenAddress = USDT` and `nativeToQuoteSwapEnabled = true` (i.e., `QuoteTokenConfiguration.nativeToQuoteSwapType != SWAP_DISABLED`)
- The token is in `TokenStatus.Tradable` (bonding curve phase)
- The BNBâ†’USDT pool used by `MULTI_DEX_ROUTER` has sufficient depth to accept the swap but returns fewer USDT than a naive 1:1 BNB-value conversion due to pool fees and slippage

**attacker-controlled inputs:**
- `params.inputToken = address(0)` (native BNB)
- `params.outputToken = <USDT-quote token address>`
- `params.inputAmount = X BNB`
- `msg.value = X BNB`
- `params.minOutputAmount` set low (or zero) to allow slippage

**call sequence:**
1. Attacker calls `Portal.swapExactInput({inputToken: address(0), outputToken: token, inputAmount: X, minOutputAmount: 0, ...})` with `msg.value = X`
2. Portal delegates to `PORTAL_TRADE_V2`
3. `PORTAL_TRADE_V2` detects `inputToken == address(0)` and `quoteTokenAddress != address(0)` â†’ triggers nativeToQuoteSwap path
4. `MULTI_DEX_ROUTER.swapExactTokensForTokens` (or `exactInputSingle`) is called with `X BNB`, returning `Y USDT` where `Y < X * price(BNB/USDT)` due to pool fees/slippage
5. **Claimed bug:** `state.reserve` is incremented by a value derived from `inputAmount` / `msg.value` (BNB-denominated) rather than the actual `Y USDT` received, overstating the reserve
6. Attacker (or a colluding seller) calls `swapExactInput` sell path, which computes sell proceeds from the inflated `state.reserve`, extracting more USDT than was actually deposited

## Why Existing Checks Fail
The `minOutputAmount` field in `ExactInputParams` guards the *final output* (meme tokens received by the buyer), not the intermediate BNBâ†’USDT conversion amount. [4](#0-3) 

If the implementation passes `amountOutMin = 0` (or a stale quote) to the router for the BNBâ†’USDT leg, slippage is unguarded at the reserve-accounting level. The `MULTI_DEX_ROUTER` interface shows `amountOutMin` is a caller-supplied parameter â€” Portal controls what value it passes. [5](#0-4) 

**Critical unknown:** The `PORTAL_TRADE_V2` source is not in the indexed repository. It is impossible to confirm from the available code whether the reserve update uses `amountOut` (actual USDT received â€” correct) or `inputAmount` / a BNB-to-USDT conversion of `msg.value` (incorrect). This is the single fact that determines whether this is a real bug or expected behavior.

## Rejection Checks
**expected behavior checked:** Normal slippage chosen by the user is rejected per rules â€” but this claim is about Portal's *internal accounting* using the wrong value for `state.reserve`, not about the user accepting bad price. If the reserve is updated with the actual `amountOut`, the invariant holds and this is a REJECT. If it uses `inputAmount`, it is a genuine accounting bug.

**prior report checked:** Not determinable from available source.

**README/NatSpec checked:** The `swapExactInput` NatSpec documents the nativeToQuoteSwap scenario but does not specify which value is used for the reserve update. [6](#0-5) 

**unsupported assumption checked:** The assumption that `state.reserve` is updated with `inputAmount` rather than `amountOut` is unverified â€” it requires reading `PORTAL_TRADE_V2` source.

**expected assertion (invariant holds â€” no bug):**
