## Verdict
REJECT

## Rejection Reason

The interface definition in `src/interfaces/IPortal.sol` defines `error SaltLockFeeMismatch(uint256 required, uint256 provided)` at line 1455, which is the exact error that would be emitted by a strict `==` equality check. The NatSpec for `lockSalt` at line 1578 states "Fee is forwarded to FEE_RECEIVER" with no mention of refund logic, consistent with an exact-match guard. The question's premise — that the check is `>=` rather than `==` — is directly contradicted by the presence of this named error in the deployed interface. The actual `lockSalt` logic lives in the `PORTAL_LAUNCHER` delegatecall target, which is not present in the indexed source files; however, the interface contract that is present provides strong evidence the implementation uses `msg.value != SALT_LOCK_FEE → revert SaltLockFeeMismatch(SALT_LOCK_FEE, msg.value)`, not a `>=` guard. Without source-level evidence that the check is weaker than `==`, the vulnerability premise is unsupported and the candidate does not meet the threshold for NEEDS_LOCAL_PROOF.
