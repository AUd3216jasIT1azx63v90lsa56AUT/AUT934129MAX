## Rejection Reason

The actual `swapExactInput` buy implementation lives in the `PORTAL_TRADE_V2` facet contract, which is **not present in the indexed source files**. The repository contains only `src/Portal.sol` (the dispatcher), `src/PortalBase.sol`, `src/PortalCommon.sol`, and interface/library files. `Portal.sol` at line 183–184 simply resolves the implementation address and `delegatecall`s to it — the entire buy logic for native-input swaps is inside the unindexed `PORTAL_TRADE_V2` contract. [1](#0-0) 

Without the `PORTAL_TRADE_V2` source, the core claim — that the implementation uses `params.inputAmount` for the curve calculation but credits `msg.value` to the reserve — cannot be verified or falsified from source. The theory is speculative with no source-level evidence to support it.

Additionally, the `buyOnCreation` NatSpec explicitly documents that `msg.value > inputAmount` is intentional and the excess is treated as a fee (not a reserve inflation bug), which is the closest documented analogue to the proposed scenario. [2](#0-1) 

A local fork test against the live BSC proxy (`0xe2ce6ab80874fa9fa2aae65d277dd6b8e65c9de0`) would be required to determine whether `swapExactInput` enforces `msg.value == params.inputAmount`, refunds excess, or exhibits the claimed mismatch — but that is outside DeepWiki's automation boundary and the source evidence needed to promote this to `NEEDS_LOCAL_PROOF` is absent.
