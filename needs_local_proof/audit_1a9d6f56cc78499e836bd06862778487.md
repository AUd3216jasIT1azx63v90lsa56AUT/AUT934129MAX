## Exact Code Path
file: `src/Portal.sol`
function: `_getSwapImplementation` / `swapExactInput`
symbols/lines:

The routing logic at lines 66–75 is the root of the question. For `TokenStatus.Staged` (and `Invalid`, `InDuel`, `Killed`), the function falls into the `else` branch and returns `PORTAL_TRADE_V2` with only a comment as the guard: [1](#0-0) 

`swapExactInput` then unconditionally delegates to that address: [2](#0-1) 

There is no status-gate at the `Portal.sol` level before the delegatecall. The only protection is whatever `PORTAL_TRADE_V2` does internally.

## Attacker Path
**preconditions:**
- A stager calls `stageNewTokenV5` for token `T`; `T` now has `TokenStatus.Staged` with `reserve = 0` and `circulatingSupply = 0`
- `commitNewTokenV5` has NOT been called, so `T` is not yet `Tradable`

**attacker-controlled inputs:**
- `ExactInputParams { inputToken: address(0), outputToken: T, inputAmount: X, minOutputAmount: 0, permitData: "" }`

**call sequence:**
1. Attacker calls `Portal.swapExactInput(params)` with `msg.value = X`
2. `_getSwapImplementation(address(0), T)` reads `state.status == TokenStatus.Staged` → falls into `else` → returns `PORTAL_TRADE_V2`
3. `_delegateToImpl(PORTAL_TRADE_V2)` executes `PORTAL_TRADE_V2.swapExactInput` in Portal's storage context
4. If `PORTAL_TRADE_V2` does not independently check `state.status == Tradable`, it proceeds against `reserve = 0` / `circulatingSupply = 0`

## Why Existing Checks Fail
`Portal.sol` performs no status assertion before the delegatecall. The comment "will fail in implementation" is developer intent, not an enforced guard. [3](#0-2) 

The `PORTAL_TRADE_V2` implementation is an immutable address set at construction time: [4](#0-3) 

Its source is **not present in this repository**. Whether it independently checks `state.status == TokenStatus.Tradable` before executing bonding-curve math against `reserve`/`circulatingSupply` cannot be confirmed from the indexed source alone. The `TokenNotTradable` error is defined in the interface: [5](#0-4) 

but no grep match for `TokenNotTradable` or a `status == Tradable` guard appears in any of the three in-scope `.sol` files (`Portal.sol`, `PortalBase.sol`, `PortalCommon.sol`). The check, if it exists, lives entirely inside the off-repo `PORTAL_TRADE_V2` facet.

## Rejection Checks
**expected behavior checked:** The comment "will fail in implementation" signals developer intent that `PORTAL_TRADE_V2` should revert — but this is unverifiable from the available source. If `PORTAL_TRADE_V2` does check status, the call reverts and there is no bug. If it does not, the invariant that Staged tokens must not be tradeable is broken.

**prior report checked:** No prior report found in the indexed corpus.

**README/NatSpec checked:** `TokenStatus.Staged` is documented as "address determined, but not yet created": [6](#0-5) 
Trading a token whose ERC-20 contract does not yet exist would violate this invariant.

**unsupported assumption checked:** The hypothesis requires that `PORTAL_TRADE_V2` omits a status check. This is plausible (the check is absent from Portal.sol) but unconfirmed without the facet source.

## Local Proof Required
**test type:** Foundry fork test against the live BSC proxy (`0xe2ce6ab80874fa9fa2aae65d277dd6b8e65c9de0`)

**test file to add:** `test/StagedTokenSwap.t.sol`

**test setup:**
1. Fork BSC at a recent block
2. Call `stageNewTokenV5` to produce a token `T` in `Staged` status (or locate an existing staged token from `FlapTokenStaged` events)
3. Confirm `getTokenV8(T).status == TokenStatus.Staged` and `reserve == 0`

**expected assertion (safe behavior):**
```solidity
vm.expectRevert(abi.encodeWithSelector(TokenNotTradable.selector, T));
portal.swapExactInput{value: 1 ether}(
    ExactInputParams(address(0), T, 1 ether, 0, "")
);
```

**failure condition (confirms vulnerability):** If the call does NOT revert with `TokenNotTradable` — or reverts with a different error (e.g., arithmetic underflow on zero reserve) after state has been partially modified — the invariant is broken. If it succeeds and returns tokens, fund extraction is confirmed. If it reverts only due to the token contract not existing (external call failure), the impact is limited to DoS, not fund extraction.
