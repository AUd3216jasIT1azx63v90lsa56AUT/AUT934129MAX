## Exact Code Path

```
} else {
```

The `else` branch is reached for any `TokenStatus` that is neither `DEX` nor `Tradable` — including `Staged`, `Invalid`, `InDuel`, and `Killed`. For a Staged token, `_getSwapImplementation` returns `PORTAL_TRADE_V2` and `swapExactInput` delegates to it. [1](#0-0) 

The `TokenStatus.Staged` enum value is defined as the fifth member (value 5), distinct from `Tradable` (value 1) and `DEX` (value 4). [2](#0-1) 

The `TokenNotTradable` error is declared in the interface but its use site is inside `PORTAL_TRADE_V2`, whose source is not present in the indexed repository. [3](#0-2) 

## Attacker Path
**preconditions:**
- SALE_FORGE has called `stageNewTokenV5` for some token `T`, setting `_packedTokenStates[T].status = TokenStatus.Staged` with `reserve = 0` and `circulatingSupply = 0`. `commitNewTokenV5` has not yet been called.
- `stageNewTokenV5` is restricted to `SALE_FORGE` (line 314), so the attacker does not stage the token — they only need a staged token to exist, which is normal protocol operation. [4](#0-3) 

**attacker-controlled inputs:**
- `params.inputToken = address(0)` (native BNB, a configured quote token)
- `params.outputToken = T` (the staged token address)
- `params.inputAmount = some BNB amount`
- `params.minOutputAmount = 0`

**call sequence:**
1. Attacker calls `swapExactInput({inputToken: address(0), outputToken: T, ...})` with BNB value.
2. `_getSwapImplementation`: `isBuy = true` (address(0) is a configured quote token), `baseToken = T`, `state.status = TokenStatus.Staged` → falls into `else` branch → returns `PORTAL_TRADE_V2`.
3. `_delegateToImpl(PORTAL_TRADE_V2)` executes the bonding-curve buy logic in the context of Portal's storage.
4. **If** `PORTAL_TRADE_V2` does not check `state.status == TokenStatus.Tradable` before computing the bonding-curve output, it will call `LibCurve` with `reserve = 0` and `circulatingSupply = 0`. The virtual-reserve bonding curve (`k = (reserve + h)^2`) still produces a non-zero token output at the initial price, minting/transferring tokens to the attacker while crediting BNB to the reserve of an uninitialized token.

## Why Existing Checks Fail
`_getSwapImplementation` explicitly defers status validation to `PORTAL_TRADE_V2` via the comment "Let PortalTradeV2 handle the error." [5](#0-4) 

There is no status guard in `swapExactInput` itself before the delegate call. The only pre-call check is `onlyIfBitFlagsSet(CB_BIT_MASK_GLOBAL_SWITCH)`, which is a global halt switch unrelated to token status. [6](#0-5) 

Whether `PORTAL_TRADE_V2` itself checks `state.status != TokenStatus.Tradable` and reverts with `TokenNotTradable` **cannot be confirmed from the indexed source** — the implementation contract source file is not present in this repository (only the ABI JSON in `out/` is available). The developer's comment implies intent to fail, but intent is not a guard.

## Rejection Checks
**expected behavior checked:** The NatSpec comment says "will fail in implementation" — this is the intended behavior. If `PORTAL_TRADE_V2` does check the status, the call reverts and there is no bug. The question is whether the implementation matches the intent.

**prior report checked:** No prior report found in the indexed files.

**README/NatSpec checked:** The `swapExactInput` NatSpec states it "supports trading tokens that is either still on the bonding curve or already listed on DEX" — Staged is explicitly neither, consistent with the invariant that Staged tokens must not be tradeable. [7](#0-6) 

**unsupported assumption checked:** The claim that `PORTAL_TRADE_V2` proceeds without a status check is unverified. The bonding-curve math with zero reserve/supply producing a non-zero output is plausible given the virtual-reserve curve design, but requires confirmation.
