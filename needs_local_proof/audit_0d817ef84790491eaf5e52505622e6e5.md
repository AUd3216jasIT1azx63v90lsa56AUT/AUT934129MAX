Both functions delegate entirely to `PORTAL_V4_CL_LOCKER` via `_delegateToImpl`: [1](#0-0) 

No `nonReentrant` modifier or reentrancy guard is present on either function in `src/Portal.sol`. A `grep` across all `src/**/*.sol` for `nonReentrant`, `ReentrancyGuard`, `_status`, and `_locked` returns zero matches for these two entry points. [2](#0-1) 

The `IV4Locker` NatSpec explicitly confirms the callback path: `collectV4Fees` routes fees through `Portal â†’ TaxProcessor â†’ distribution`, and `addV4LPLiquidity` is documented as "Called by TaxProcessor to reinvest LP-share tax revenue."

## Attacker Path
**preconditions:**
- A V7/V4-style token has been launched and has accrued LP fees in its GoPlus/UNCX locker position.
- The TaxProcessor for that token has `lpBps > 0` (LP reinvestment share enabled), meaning it will call `addV4LPLiquidity` during distribution.
- No reentrancy lock is set in the `PORTAL_V4_CL_LOCKER` delegatecall target before the TaxProcessor callback.

**attacker-controlled inputs:**
- The attacker is an unprivileged keeper; `collectV4Fees(token)` is permissionless per NatSpec.

**call sequence:**
1. Attacker calls `Portal.collectV4Fees(token)`.
2. Portal `delegatecall`s into `PORTAL_V4_CL_LOCKER`, which collects fees from the V4 pool and calls TaxProcessor for distribution.
3. TaxProcessor, during its LP-share distribution step, calls back `Portal.addV4LPLiquidity(token, tokenAmount, quoteAmount)`.
4. Portal `delegatecall`s into `PORTAL_V4_CL_LOCKER` again â€” reentrant into the same proxy storage context â€” and increases the locker position using tokens already accounted for in step 2.
5. If the locker's accounting state (e.g., tracked fee balances or position amounts) was not finalized before the TaxProcessor callback, the same fee tokens are counted twice, inflating the LP position beyond the net new fees collected.

## Why Existing Checks Fail
No reentrancy guard is visible in `src/Portal.sol` on either `collectV4Fees` or `addV4LPLiquidity`. [1](#0-0)  The actual accounting logic lives entirely inside `PORTAL_V4_CL_LOCKER`, whose source is **not present** in the indexed repository (`src/` contains only `Portal.sol`, `PortalBase.sol`, `PortalCommon.sol`, and interface/library files).  Without that source, it is impossible to confirm whether the locker sets a reentrancy lock before invoking TaxProcessor, or whether fee-balance state is committed before or after the callback. The NatSpec confirms the callback path exists and is expected, but does not document any guard against reentrant `addV4LPLiquidity` calls during an active `collectV4Fees` execution.

## Rejection Checks
**expected behavior checked:** The TaxProcessorâ†’`addV4LPLiquidity` callback is documented expected behavior. The question is whether the ordering of state updates vs. the callback creates a double-count â€” that is not documented as expected.

**prior report checked:** No prior report found in the indexed source.

**README/NatSpec checked:** NatSpec confirms the callback path but does not document a reentrancy guard or commit-before-callback ordering guarantee.

**unsupported assumption checked:** The theory requires that `PORTAL_V4_CL_LOCKER` does not finalize fee accounting before calling TaxProcessor. This is the unverifiable assumption â€” the locker source is absent from the repo index.
