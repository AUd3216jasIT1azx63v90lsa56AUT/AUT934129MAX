## Attacker Path

**preconditions:**
- Protocol is not halted (`bitFlags & CB_BIT_MASK_GLOBAL_SWITCH != 0`).
- A target token exists with `status == TokenStatus.DEX` (already graduated).
- `PORTAL_DEX_ROUTER` implementation does **not** enforce `DEX_ROUTER_MANAGER_ROLE` (the hypothesis to be tested).

**attacker-controlled inputs:**
- `token`: address of any DEX-graduated token whose `_dexPools` entry the attacker wants to overwrite.
- `poolInfo.pool`: attacker-controlled contract that mimics a DEX pool but routes swap proceeds to the attacker.

**call sequence:**
1. Attacker calls `Portal.updateTokenPoolInfo(token, {pool: attackerPool, fee: ..., poolType: V2, ...})` from an unprivileged EOA.
2. Portal passes `onlyIfBitFlagsSet(CB_BIT_MASK_GLOBAL_SWITCH)` (global switch is on).
3. `_delegateToImpl(PORTAL_DEX_ROUTER)` executes in Portal's storage context.
4. If PORTAL_DEX_ROUTER has no role check, `_dexPools[token]` is overwritten with `attackerPool`.
5. Any subsequent `swapExactInput` call for that token routes through `_getSwapImplementation` → `PORTAL_DEX_ROUTER` → reads `_dexPools[token]` → calls `attackerPool`, draining quote tokens from the swap.

## Why Existing Checks Fail

`Portal.sol`'s `updateTokenPoolInfo` applies only `onlyIfBitFlagsSet(CB_BIT_MASK_GLOBAL_SWITCH)`, which is a liveness guard, not an access-control guard. [3](#0-2) [4](#0-3) 

The `DEX_ROUTER_MANAGER_ROLE` constant is declared in `src/interfaces/IPortal.sol` (2 occurrences), confirming the role is part of the intended design. However, the role enforcement is **not** present in `Portal.sol` itself — it must live entirely inside the `PORTAL_DEX_ROUTER` delegatecall target. The source file for `PORTAL_DEX_ROUTER` is **not** among the verified source files (`src/Portal.sol`, `src/PortalBase.sol`, `src/PortalCommon.sol`, and the listed interfaces/libraries). This means the presence or absence of the role check in the implementation facet cannot be confirmed from the indexed source alone. [5](#0-4) 

The consistent pattern in this codebase is that Portal.sol is a thin dispatcher and all substantive access control lives in the facet (e.g., `PORTAL_TWEAK`, `PORTAL_ROLLER`). If the PORTAL_DEX_ROUTER facet omits the `hasRole(DEX_ROUTER_MANAGER_ROLE, msg.sender)` check, the invariant is broken.

## Rejection Checks

**expected behavior checked:** The `DEX_ROUTER_MANAGER_ROLE` is defined in the interface, indicating role-gated writes to `_dexPools` are the intended design. Whether the implementation enforces it is the open question. [6](#0-5) 

**prior report checked:** No prior report found in the indexed source.

**README/NatSpec checked:** No NatSpec on `updateTokenPoolInfo` in Portal.sol beyond `@inheritdoc IPortalDexRouter`. The `TokenPoolInfoUpdated` event NatSpec does not indicate public access.

**unsupported assumption checked:** The attack requires the PORTAL_DEX_ROUTER facet to lack the role check. This is the unverifiable assumption — it is plausible given the missing source, but not confirmed.

    function _checkBitFlags(uint256 mask) internal view returns (bool) {
        // We only check the bits in the mask are set
        // And ignore the bits that are not in the mask
        //
        //  -  mask => with only checking bits set
        //  -  mask ^ bitFlags =>
        //          XOR: all checking bits should be unset, if
        //          they are set in the bitFlags.
        //  - _ & mask, ignore the bits that are not in the mask
        //
        return (mask ^ bitFlags) & mask == 0;
    }
