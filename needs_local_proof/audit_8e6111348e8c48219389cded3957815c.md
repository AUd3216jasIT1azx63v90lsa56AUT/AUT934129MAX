## Attacker Path
**preconditions:** Any token in `TokenStatus.Tradable` (bonding curve phase) with `quoteToken = NATIVE_GAS_TOKEN` (address(0) configured in `_quoteTokenConfigurations`). Attacker optionally pre-buys tokens at a lower price before the inflation step.

**attacker-controlled inputs:**
- `params.inputToken = address(0)` (native buy)
- `params.outputToken = <target token>`
- `params.inputAmount = X` (small)
- `msg.value = Y` where `Y >> X`

**call sequence:**
1. (Optional) Attacker calls `swapExactInput` normally to accumulate tokens at current price.
2. Attacker calls `swapExactInput({inputToken: address(0), outputToken: token, inputAmount: X, ...})` with `msg.value = Y > X`.
3. If `PORTAL_TRADE_V2` uses `params.inputAmount` for the curve calculation (minting tokens for X) but the reserve is updated by `msg.value` (Y), the stored reserve grows by Y while only X-worth of tokens are minted.
4. Attacker sells pre-accumulated tokens against the inflated reserve, extracting more BNB than the curve should allow.

## Why Existing Checks Fail
The `swapExactInput` dispatcher in `src/Portal.sol` performs no `msg.value` vs `inputAmount` reconciliation before delegating — it is a pure passthrough via `_delegateToImpl(PORTAL_TRADE_V2)`. [1](#0-0) 

The `_getSwapImplementation` function only routes based on token status; it does not validate payment amounts. [2](#0-1) 

The `PORTAL_TRADE_V2` implementation is not present in the indexed source files, so whether it enforces `msg.value == params.inputAmount` for native buys, or whether it uses `params.inputAmount` for the curve but credits `msg.value` to the reserve, cannot be determined from source alone.

The `InsufficientEth` custom error exists in the interface, confirming the implementation does check `msg.value >= inputAmount`: [3](#0-2) 

However, this only establishes a lower-bound check. It does not confirm whether excess `msg.value` above `inputAmount` is (a) rejected, (b) kept as a fee without updating the reserve, or (c) credited to the reserve while only `inputAmount` is used for the curve calculation (the bug scenario).

Notably, `buyOnCreation` explicitly documents that excess ETH is kept as a fee and not refunded: [4](#0-3) 

No equivalent NatSpec exists for `swapExactInput`, leaving the excess-ETH handling undocumented and unverifiable from the available source.

The reserve is stored separately from the contract's actual ETH balance via `_setTokenReserve`: [5](#0-4) 

If `PORTAL_TRADE_V2` calls `_setTokenReserve(token, oldReserve + msg.value - fee)` instead of `_setTokenReserve(token, oldReserve + inputAmount - fee)`, the invariant is broken: the stored reserve exceeds the ETH actually accounted for by the curve calculation, inflating the bonding curve price for all subsequent trades.

## Rejection Checks

**README/NatSpec checked:** `swapExactInput` NatSpec contains no mention of excess ETH handling. `buyOnCreation` NatSpec explicitly documents it — the absence for `swapExactInput` is notable.

**unsupported assumption checked:** The core assumption — that `PORTAL_TRADE_V2` uses `inputAmount` for curve math but `msg.value` for reserve update — is unverifiable without the implementation source. The hypothesis is structurally plausible but not confirmed.

    function _getSwapImplementation(address inputToken, address outputToken)
        internal
        view
        returns (address implementation)
    {
        // Determine if this is a buy or sell
        bool isBuy = _quoteTokenConfigurations[inputToken].enabled == 1;
        address baseToken = isBuy ? outputToken : inputToken;

        // Get token state to check status
        PackedTokenStateV2 memory state = _getTokenState(baseToken);

        if (state.status == TokenStatus.DEX) {
            // Token is listed on DEX, use PortalDexRouter
            return PORTAL_DEX_ROUTER;
        } else if (state.status == TokenStatus.Tradable) {
            // Token is still in bonding curve, use PortalTradeV2
            return PORTAL_TRADE_V2;
        } else {
            // Invalid or other status, will fail in implementation
            return PORTAL_TRADE_V2; // Let PortalTradeV2 handle the error
        }
    }
