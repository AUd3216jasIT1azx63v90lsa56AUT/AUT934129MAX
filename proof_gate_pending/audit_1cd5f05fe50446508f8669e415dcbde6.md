## Attacker Path
**preconditions:**
- A token is in `TokenStatus.Tradable` with `circulatingSupply` just below `dexSupplyThresh`
- Attacker controls a receiver contract that can re-enter `swapExactInput` during an ETH refund or ERC20 callback triggered inside the `PORTAL_TRADE_V2` delegatecall (e.g., during the `luanchToDEX` external call to the migrator)

**attacker-controlled inputs:**
- `ExactInputParams.inputToken = address(0)`, `outputToken = <token>`, `inputAmount` sized to push `circulatingSupply >= dexSupplyThresh`
- Attacker's `recipient` is a contract with a `receive()` or ERC20 callback that re-calls `swapExactInput`

**call sequence:**
1. Attacker calls `swapExactInput` with enough BNB to trigger migration
2. Inside `PORTAL_TRADE_V2` delegatecall: buy executes, `circulatingSupply >= dexSupplyThresh` is detected
3. If `_setTokenStatus(token, TokenStatus.DEX)` is called **after** the external `luanchToDEX` call (or if any ETH/token transfer to the attacker's address occurs before status flip), attacker's contract re-enters `swapExactInput`
4. Re-entrant call: `_getSwapImplementation` reads `state.status` — if still `Tradable`, routes to `PORTAL_TRADE_V2` again
5. Attacker buys additional tokens at bonding-curve price while migration reserve is being consumed

## Why Existing Checks Fail

**No reentrancy guard anywhere in source:** A grep across all `src/**/*.sol` files finds zero occurrences of `nonReentrant`, `ReentrancyGuard`, `_status`, or any equivalent lock. [1](#0-0)  The `swapExactInput` entry point carries only `onlyIfBitFlagsSet(CB_BIT_MASK_GLOBAL_SWITCH)`, which is a liveness flag, not a reentrancy lock.

**Routing guard is ordering-dependent:** `_getSwapImplementation` reads `state.status` from storage at the start of each top-level `swapExactInput` call. [5](#0-4)  If `_setTokenStatus(token, TokenStatus.DEX)` is written to storage **before** the external `luanchToDEX` call, a re-entrant call would see `TokenStatus.DEX` and be routed to `PORTAL_DEX_ROUTER` (which would fail on an unfunded pool — effectively blocking the attack). However, if the status flip happens **after** the external migrator call, the re-entrant call still sees `TokenStatus.Tradable` and is routed to `PORTAL_TRADE_V2`, opening the window. The ordering is inside `PORTAL_TRADE_V2`, an immutable delegatecall target whose source is not in the indexed files — this is the unresolved crux.

**`_setTokenStatus` is a plain storage write with no lock semantics:** [3](#0-2)  It provides a CEI-style guard only if called strictly before any external interaction in the migration path.

## Rejection Checks
- **expected behavior checked:** Migration triggering on threshold crossing is documented; the question is whether the status flip precedes the external call.
- **prior report checked:** No prior report found in indexed files.
- **README/NatSpec checked:** `IPortalMigrator.luanchToDEX` NatSpec notes it "may be delegated called from a payable function" — acknowledges external call context but no reentrancy note. [6](#0-5) 
- **unsupported assumption checked:** The attack requires the migrator to make a callback or ETH transfer to an attacker-controlled address before the status flip. This is plausible (BNB refund to buyer, or ERC20 `transfer` to recipient) but unconfirmed without the `PORTAL_TRADE_V2` source.

        if (impl == address(0)) {
            revert FeatureDisabled();
        }
        assembly {
            calldatacopy(0, 0, calldatasize())
            let result := delegatecall(gas(), impl, 0, calldatasize(), 0, 0)
            returndatacopy(0, 0, returndatasize())
            switch result
            case 0 { revert(0, returndatasize()) }
            default { return(0, returndatasize()) }
        }
    }

        if (token != address(uint160(uint256(uint160(token))))) revert DirtyBits();
        uint256 slot;
