The actual fee-collection and liquidity-addition logic lives in `PORTAL_V4_CL_LOCKER` (referenced in comments as `PortalUniV4Locker.sol`), which is **not present in the indexed source files**. The `src/` directory contains only `Portal.sol`, `PortalBase.sol`, `PortalCommon.sol`, and the interface/library files. [2](#0-1) 

## Attacker Path

**preconditions:**
- A V7 token with `LP_BPS` fee type has been launched and has accumulated LP fees in the GoPlus/UNCX locker.
- The TaxProcessor for that token is configured to call back `Portal.addV4LPLiquidity` during its distribution phase (this is the documented design per NatSpec).
- `PORTAL_V4_CL_LOCKER` implementation does **not** apply a reentrancy lock or CEI ordering that prevents `addV4LPLiquidity` from executing while `collectV4Fees` is mid-flight.

**attacker-controlled inputs:**
- `token` address passed to `collectV4Fees(token)` — permissionless, anyone can call.

**call sequence:**
1. Attacker calls `Portal.collectV4Fees(token)` → delegatecall to `PORTAL_V4_CL_LOCKER`.
2. Locker collects fees from the pool position; tokens land in Portal's balance.
3. Locker calls `TaxProcessor.distribute(...)` to split fees per `feeConfigs`.
4. TaxProcessor, handling the `LP_BPS` slot, calls back `Portal.addV4LPLiquidity(token, tokenAmt, quoteAmt)` → delegatecall to `PORTAL_V4_CL_LOCKER`.
5. If the locker has not yet updated its internal accounting (or has no reentrancy lock), `addV4LPLiquidity` increases LP using tokens that `collectV4Fees` still considers unspent.
6. Control returns to `collectV4Fees`; the same token amounts are processed a second time, either double-adding to LP or consuming tokens from Portal's pre-existing balance.

## Why Existing Checks Fail

**No reentrancy guard at the Portal dispatcher level.** Neither `collectV4Fees` nor `addV4LPLiquidity` in `src/Portal.sol` carries a `nonReentrant` modifier or any lock. A grep across all `src/**/*.sol` files confirms zero matches for `nonReentrant`, `ReentrancyGuard`, `_locked`, or `_reentrancy`. [1](#0-0) 

**Whether the locker implementation itself has a guard is unknown** — `PortalUniV4Locker.sol` is not in the indexed source. The NatSpec for `addV4LPLiquidity` explicitly documents it as "Called by TaxProcessor to reinvest LP-share tax revenue," confirming the designers anticipated this callback path, but does not state whether a reentrancy lock is present inside the locker. [3](#0-2) 

**The TaxProcessor callback is the designed flow.** The `LP_BPS` fee type in `FeeConfig` is specifically intended to route collected fees back through `addV4LPLiquidity`. This means the re-entrant call is not an edge case — it is the normal execution path for every `collectV4Fees` call on a token with `LP_BPS` configured. [4](#0-3) 

## Rejection Checks

**expected behavior checked:** The TaxProcessor→`addV4LPLiquidity` callback is explicitly documented as the designed behavior for `LP_BPS` fee tokens. Whether the accounting inside the locker correctly handles this re-entrant call (CEI, checks-effects-interactions) cannot be determined without `PortalUniV4Locker.sol`.

**prior report checked:** No prior report found in the indexed source.

**README/NatSpec checked:** NatSpec confirms `collectV4Fees` is permissionless and that `addV4LPLiquidity` is the TaxProcessor callback target. No guard is mentioned.

**unsupported assumption checked:** The attacker profit path is the weakest link. Calling `collectV4Fees` does not send funds directly to the attacker. For extraction to occur, the attacker would need to be the token beneficiary (or marketing address) who can later claim from the inflated LP position, or the double-counting must drain Portal's pre-existing token/quote balance in a way that is claimable. This needs to be confirmed in the locker implementation.

## Local Proof Required

**test type:** Foundry fork test on BSC mainnet (block ≥ 108382650)

**test file to add:** `test/fork/CollectV4FeesReentrancy.t.sol`

**test setup:**
1. Fork BSC at the latest block; identify a live V7 token with `LP_BPS` fee config and accumulated locker fees (use `V4LPFeesCollected` events to find a candidate).
2. Deploy a mock `TaxProcessor` that, during `distribute()`, calls back `Portal.addV4LPLiquidity(token, tokenAmt, quoteAmt)` with the same amounts being distributed.
3. Temporarily replace the token's TaxProcessor address with the mock (requires admin role — if not possible, use a fork-state override via `vm.store`).
4. Record Portal's token and quote balances before the call.
5. Call `Portal.collectV4Fees(token)` from an unprivileged address.

**expected assertion:**
- If vulnerable: Portal's token/quote balance decreases by more than the net fees collected in this call, OR the LP position's liquidity increases by more than the collected fee amount, OR the call succeeds without reverting and token balance deltas are inconsistent with a single fee collection.
- If safe: the call reverts with a reentrancy error, or balances reconcile exactly with one fee collection cycle.

**failure condition:** The test is inconclusive without `PortalUniV4Locker.sol` source. If the locker applies a storage-based reentrancy lock (e.g., a `_collecting` flag set before the TaxProcessor call and cleared after), the callback to `addV4LPLiquidity` will revert and the bug does not exist. The fork test is the only way to resolve this given the missing implementation source.
