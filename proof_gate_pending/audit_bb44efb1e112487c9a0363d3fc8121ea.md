## Rejection Reason

The actual permit-handling logic for `swapExactInput` lives in the `PORTAL_TRADE_V2` delegatecall facet, which is not present in the indexed source files (`src/Portal.sol`, `src/PortalBase.sol`, `src/PortalCommon.sol`). [1](#0-0)  The Portal entry point does nothing but forward the full calldata via `_delegateToImpl(implementation)` — all permit decoding and `transferFrom` sequencing is inside the unindexed facet. [2](#0-1) 

A grep across every `.sol` file under `src/` for `permitData`, `permit(`, `_usePermit`, and `_applyPermit` returns **zero matches** outside of `src/interfaces/IPortal.sol` (struct definitions only). [3](#0-2)  There is therefore no source-level evidence of how the facet sequences the `permit()` call relative to `transferFrom()`.

The OZ `IERC20PermitUpgradeable` documentation bundled in this very repo explicitly states the recommended pattern is `permit(msg.sender, address(this), ...)` so that the signer and the caller are the same address, making cross-user replay impossible by construction. [4](#0-3)  Without source-level evidence that the facet deviates from this pattern (e.g., decoding an arbitrary `owner` from `permitData` and then passing that same `owner` to `transferFrom`), the attack is purely speculative. Promoting it to `NEEDS_LOCAL_PROOF` or higher requires at minimum the facet source or bytecode disassembly showing the deviant sequencing.
