## Attacker Path
**preconditions:** A TOKEN_TAXED or TOKEN_TAXED_V2 token exists on the bonding curve (pre-DEX graduation). `enableTaxOnBondingCurve()` returns true. The attacker is an unprivileged EOA.

**attacker-controlled inputs:**
- `params.inputToken = address(0)` (native BNB)
- `params.outputToken = <taxToken>`
- `params.inputAmount = 1 BNB`
- `msg.value = 1 BNB`

**call sequence:**
1. Attacker calls `swapExactInput({inputToken: 0, outputToken: taxToken, inputAmount: 1e18, minOutputAmount: 0, permitData: ""})` with `msg.value = 1e18`.
2. Portal delegates to the swap facet. Facet computes `taxAmount = 1e18 * taxRate / 10000` and `protocolFee = 1e18 * FLAP_BUY_FEE / 10000`.
3. **Claimed bug:** Curve calculation uses `curveInput = 1e18 - taxAmount` (tax-deducted), minting tokens consistent with that smaller input. But the reserve is updated with the full `1e18` (or `1e18 - protocolFee` without subtracting tax), so `reserve` grows by more than `curveInput`.
4. Attacker sells the received tokens back via `swapExactInput({inputToken: taxToken, outputToken: 0, ...})`. The sell path computes `estimateReserve(curve, newSupply)` using the inflated reserve, returning more BNB than the curve formula warrants.
5. Net result: attacker extracts BNB from the proxy balance beyond what they paid minus fees. [4](#0-3) [5](#0-4) 

## Why Existing Checks Fail
The `minOutputAmount` slippage guard does not protect against reserve over-inflation — it only prevents the attacker from receiving fewer tokens than expected, not from receiving tokens priced against an inflated reserve. The `estimateReserve` round-up in `LibCurve` (line 66) is designed to protect the protocol on sells, but if the reserve itself is inflated by the uncredited tax amount, the round-up direction does not compensate. No re-entrancy guard or reserve reconciliation check is visible in the indexed source that would catch a mismatch between `msg.value` credited to the reserve and the tax-deducted amount used for curve pricing. [6](#0-5) 

## Rejection Checks
**expected behavior checked:** Tax tokens are documented to collect tax on the bonding curve (`TaxOnBondingCurvePaid` event, `enableTaxOnBondingCurve()` view). The question is whether the reserve accounting correctly excludes the tax portion — this is NOT documented as intentional behavior where reserve absorbs the tax. [7](#0-6) 

**prior report checked:** No prior report matching this exact reserve-inflation path is visible in the indexed source or NatSpec. 

**README/NatSpec checked:** No NatSpec on the swap facet (not in indexed source) explicitly states that tax is credited to the reserve rather than forwarded to the tax splitter. [8](#0-7) 

**unsupported assumption checked:** The critical assumption — that the reserve update uses `msg.value` rather than `msg.value - taxAmount` — cannot be confirmed from the indexed source alone because the swap facet contract is not in the verified source file list. This is the gating uncertainty.

**expected assertion (if bug is real):**

```text
    /// @notice estimate the reserve given the supply
    /// @dev This function returns a roundup value, because we want the following invariant to hold:
    ///         currReserve >= estimateReserve_without_roudup(currSupply)
    ///
    ///      This function is used when a user wants to sell tokens, a rounded up value
    ///      is more favorable to the protocol.
    function estimateReserve(Curve memory curve, uint256 supply) internal pure returns (uint256 reserve) {
        if (supply > TOTAL_SUPPLY) {
            revert SupplyExceedsTotalSupply(supply);
        }

        // eth = k/(h + 1e9 - s) - r
        // Round up for protocol safety when selling
        reserve = FixedPointMathLib.divWadUp(curve.k, TOTAL_SUPPLY + curve.h - supply) - curve.r;
    }
```
