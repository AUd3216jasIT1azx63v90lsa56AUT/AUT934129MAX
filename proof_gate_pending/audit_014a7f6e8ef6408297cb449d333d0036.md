## Attacker Path
**preconditions:**
- Attacker is the registered beneficiary of a launched token that has accumulated LP fees (post-DEX migration, `lpLocks` populated).
- Attacker deploys a contract with a `receive()` fallback that re-enters `claim(token)`.

**attacker-controlled inputs:**
- `token` address (the token for which attacker is beneficiary).

**call sequence:**
1. Attacker contract calls `Portal.claim(token)`.
2. Portal dispatches via `delegatecall` to `PORTAL_ROLLER`.
3. PORTAL_ROLLER reads claimable LP fee amount.
4. *(If CEI is violated)* PORTAL_ROLLER sends BNB/quote token to attacker before zeroing the claimable balance.
5. Attacker's `receive()` re-enters `Portal.claim(token)`.
6. PORTAL_ROLLER reads the same (not-yet-zeroed) claimable amount and sends again.
7. Attacker drains double (or more) the entitled LP fees.

## Why Existing Checks Fail
**No reentrancy guard anywhere in `src/**/*.sol`:** A grep for `nonReentrant`, `ReentrancyGuard`, `_status`, `_ENTERED`, and `_NOT_ENTERED` across all `src/` Solidity files returns zero matches. [1](#0-0)  Neither `claim` nor `delegateClaim` carry any modifier beyond `onlyIfBitFlagsSet(CB_BIT_MASK_GLOBAL_SWITCH)`, which is a simple bitflag read and provides no reentrancy protection. [5](#0-4) 

**Critical gap — PORTAL_ROLLER source not indexed:** The PORTAL_ROLLER implementation is an immutable address injected at construction time. [4](#0-3)  Its source file (`PortalRoller.sol` or equivalent) is **not present** in the indexed repository. Therefore it is impossible to statically confirm whether the implementation follows CEI (clear claimable → transfer) or the vulnerable pattern (transfer → clear claimable). This is the single blocking unknown.

**`_delegateToImpl` is a raw assembly delegatecall** with no guard wrapping: [3](#0-2)  Any reentrancy protection must therefore live entirely inside the PORTAL_ROLLER implementation itself.

## Rejection Checks
**expected behavior checked:** Claiming LP fees for a beneficiary is a value-transfer operation; reentrancy protection is not "expected" to be absent — this is a genuine concern, not documented protocol behavior.

**prior report checked:** No prior report or known-issues file found in the indexed repo.

**README/NatSpec checked:** `IRoller.claim` NatSpec says "Only the beneficiary of the token can call this function" — no mention of reentrancy safety. [6](#0-5) 

**unsupported assumption checked:** The assumption that PORTAL_ROLLER does *not* clear before transfer is unverified — this is exactly what local proof must determine.

## Local Proof Required
**test type:** Foundry fork test on BSC mainnet (block ≥ 108382650).

**test file to add:** `test/ClaimReentrancy.t.sol`

**test setup:**
1. Fork BSC at latest block; use proxy `0xe2ce6ab80874fa9fa2aae65d277dd6b8e65c9de0`.
2. Find a token with a live `BeneficiaryClaimed`-emitting history (event-derived key from live context) and non-zero claimable LP fees via `getLocks(token)` / `collectV4Fees` preview.
3. Deploy attacker contract: beneficiary address, `receive()` re-enters `Portal.claim(token)` once, records cumulative ETH received.
4. Use `vm.prank` / `vm.store` to set `_tokenBeneficiaries[token] = attackerContract`.

**expected assertion:**
```
assertEq(attackerContract.totalReceived(), singleClaimAmount);
// PASS  → implementation is safe (CEI or internal guard)
// FAIL  → double-claim confirmed, HIGH_CONFIDENCE_CANDIDATE
```

**failure condition:** If `totalReceived > singleClaimAmount`, the implementation transfers before clearing, reentrancy is exploitable, and the finding is promotable to HIGH_CONFIDENCE_CANDIDATE with concrete BNB extraction from the live 1361 BNB proxy balance.
