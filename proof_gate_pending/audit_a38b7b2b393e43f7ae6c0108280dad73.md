## Exact Code Path
file: `src/Portal.sol`
function: `setTokenBeneficiary`
symbols/lines: lines 351–360

The Portal dispatcher has **no access control modifier** at the dispatch layer — it unconditionally delegates to `PORTAL_ROLLER`:

```solidity
function setTokenBeneficiary(
    address,  /*token*/
    address   /* newBeneficiary */
)
    external
    override
{
    _delegateToImpl(PORTAL_ROLLER);   // no onlyRole guard here
}
``` [1](#0-0) 

The interface NatSpec documents the intent as admin-only:

> "Allows the **default admin** to change the beneficiary of a token" [2](#0-1) 

The `_tokenBeneficiaries` mapping (slot 169) is what `claim()` reads to determine who receives LP fees: [3](#0-2) 

The `claim()` NatSpec confirms fees go to the beneficiary and only the beneficiary may call it: [4](#0-3) 

## Attacker Path
**preconditions:**
- A token exists with a legitimate beneficiary (e.g., the original creator).
- The attacker is any external caller (does not need to be the current beneficiary — the question assumes they are, but the real question is whether *any* caller can invoke this).
- The PORTAL_ROLLER implementation either (a) has no `onlyRole(DEFAULT_ADMIN_ROLE)` guard, or (b) allows the current beneficiary to change the beneficiary.

**attacker-controlled inputs:**
- `token` = address of a launched token with accumulated LP fees
- `newBeneficiary` = attacker-controlled address

**call sequence:**
1. Attacker calls `Portal.setTokenBeneficiary(token, attackerAddress2)` — dispatched via `_delegateToImpl(PORTAL_ROLLER)`.
2. If PORTAL_ROLLER lacks the access control check, `_tokenBeneficiaries[token]` is overwritten with `attackerAddress2`.
3. Attacker calls `Portal.claim(token)` from `attackerAddress2` — LP fees are sent to `attackerAddress2`.

## Why Existing Checks Fail
The Portal.sol dispatcher applies **no `onlyRole` modifier** to `setTokenBeneficiary`. All other admin-only functions in Portal.sol (e.g., `setQuoteTokenConfiguration`, `setFeeExemption`, `updateTaxTokenAddresses`) also lack dispatch-level guards and rely entirely on the delegatecall target to enforce access control. Whether PORTAL_ROLLER enforces `DEFAULT_ADMIN_ROLE` cannot be confirmed from the indexed source: the PORTAL_ROLLER implementation contract source is **not present** in the repository's `src/` directory (the verified source identity lists only `Portal.sol`, `PortalBase.sol`, `PortalCommon.sol`, and interface/library files). The access control check, if it exists, lives entirely inside the external PORTAL_ROLLER bytecode. [1](#0-0) 

## Rejection Checks
**expected behavior checked:** The interface NatSpec explicitly says "default admin" only — so if any non-admin can call this, it violates the documented invariant.

**prior report checked:** No prior report found in the indexed corpus.

**README/NatSpec checked:** NatSpec on `IRoller.setTokenBeneficiary` states admin-only. The `BeneficiaryChanged` event exists confirming the state change is observable.

**unsupported assumption checked:** The attack does not require oracle manipulation, malicious tokens, or admin key compromise — it only requires the PORTAL_ROLLER implementation to be missing or weakening the access control check.

## Local Proof Required
**test type:** Foundry fork test against the live BSC proxy (`0xe2ce6ab80874fa9fa2aae65d277dd6b8e65c9de0`)

**test file to add:** `test/SetTokenBeneficiaryAccessControl.t.sol`

**test setup:**
1. Fork BSC at a recent block.
2. Identify a live token with a known beneficiary (from `BeneficiaryChanged` or `TokenCreated` events).
3. Use a fresh EOA (no roles) as the attacker.

**expected assertion (if vulnerable):**
```solidity
// Non-admin can redirect beneficiary
vm.prank(attacker);
portal.setTokenBeneficiary(token, attacker);
assertEq(portal.getTokenBeneficiary(token), attacker); // succeeds → VULNERABLE

// Then claim from attacker
vm.prank(attacker);
(uint256 tok, uint256 eth) = portal.claim(token);
assertGt(eth, 0); // attacker extracted LP fees
```

**expected assertion (if safe):**
```solidity
vm.prank(attacker);
vm.expectRevert(); // should revert with AccessControl error
portal.setTokenBeneficiary(token, attacker);
```

**failure condition:** If the non-admin call to `setTokenBeneficiary` succeeds and `_tokenBeneficiaries[token]` is updated, the vulnerability is confirmed. The test cannot be resolved from source alone because the PORTAL_ROLLER implementation bytecode is not available in the indexed repository.
