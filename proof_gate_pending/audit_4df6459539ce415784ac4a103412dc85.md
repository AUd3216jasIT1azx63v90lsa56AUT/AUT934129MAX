## Exact Code Path
file: `src/Portal.sol`
function: `swapExactInput`
symbols/lines: lines 174–185 (dispatch), `ExactInputParams.inputAmount` vs `msg.value` in the delegated `PORTAL_TRADE_V2` implementation

The `swapExactInput` entry point in `src/Portal.sol` is `payable` and accepts `ExactInputParams` which carries a separate `inputAmount` field. [1](#0-0)  For a native buy (`inputToken = address(0)`), both `msg.value` and `params.inputAmount` are available to the implementation. The actual bonding-curve accounting lives in `PORTAL_TRADE_V2`, which is reached via `_delegateToImpl` and is **not present in the indexed source files** of this repo. [2](#0-1) 

The `ExactInputParams` struct explicitly separates `inputAmount` from the payable ETH channel. [3](#0-2) 

Contrast with `buyOnCreation`, whose NatSpec explicitly documents that `msg.value > inputAmount` is intentional and the excess is kept as fee — no equivalent documentation exists for `swapExactInput`. [4](#0-3) 

The bonding curve reserve field is a `uint128` in `PackedTokenStateV2` and is the sole source of truth for price. [5](#0-4)  The curve invariant `(1e9 + h - s) * (eth + r) = k` means any reserve inflation directly raises the price for all subsequent buyers. [6](#0-5) 

## Attacker Path
**preconditions:** Any token with `TokenStatus.Tradable` (still on bonding curve, native quote token). No role or privilege required — `swapExactInput` is permissionless.

**attacker-controlled inputs:**
- `params.inputToken = address(0)` (native buy)
- `params.outputToken = <target token>`
- `params.inputAmount = X` (small declared amount)
- `msg.value = Y` where `Y >> X`

**call sequence:**
1. Attacker calls `Portal.swapExactInput({inputToken: address(0), outputToken: token, inputAmount: X, minOutputAmount: 0, permitData: ""})` with `msg.value = Y`.
2. Portal dispatches to `PORTAL_TRADE_V2` via `delegatecall`.
3. **If** `PORTAL_TRADE_V2` uses `inputAmount` for the curve calculation (tokens minted = `f(X)`) but updates the reserve by `msg.value` (reserve += `Y`): reserve grows by `Y` while circulating supply grows only by `f(X)`.
4. Bonding curve price is now inflated. Attacker (or any subsequent buyer) pays more per token than the reserve/supply ratio warrants.
5. Attacker can exploit this to manipulate price before a large sell, or to grief other buyers.

## Why Existing Checks Fail
The `swapExactInput` entry point applies only `onlyIfBitFlagsSet(CB_BIT_MASK_GLOBAL_SWITCH)` — no check that `msg.value == params.inputAmount` for native buys. [7](#0-6)  Whether `PORTAL_TRADE_V2` enforces this equality is unknown from the indexed source. The `buyOnCreation` NatSpec explicitly permits `msg.value > inputAmount` as a fee mechanism, suggesting the codebase has precedent for not enforcing equality — but `swapExactInput` has no such documented intent. [8](#0-7) 

**Critical limitation:** The `PORTAL_TRADE_V2` module is not present in the indexed source files of this repository. The source inventory lists only `src/Portal.sol`, `src/PortalBase.sol`, `src/PortalCommon.sol`, and the interface/library files.  The actual reserve-update logic — whether it uses `inputAmount` or `msg.value` — cannot be confirmed from available code. This is the blocking gap.

## Rejection Checks
**expected behavior checked:** `buyOnCreation` documents `msg.value > inputAmount` as intentional fee behavior. No equivalent NatSpec exists for `swapExactInput`, so this is not clearly expected behavior for the swap path. [9](#0-8) 

**prior report checked:** Not determinable from indexed source alone.

**README/NatSpec checked:** `swapExactInput` NatSpec describes scenarios but says nothing about excess `msg.value` handling. [10](#0-9) 

**unsupported assumption checked:** The entire hypothesis rests on `PORTAL_TRADE_V2` using `inputAmount` for curve math but `msg.value` for reserve update. This split behavior is the unverified assumption — it could equally use `inputAmount` for both (no bug, excess ETH silently donated) or `msg.value` for both (no invariant break). Cannot resolve without the implementation source.
