## Attacker Path
**preconditions:**
- A token is in `TokenStatus.Tradable` with `circulatingSupply` just below `dexSupplyThresh`
- Attacker controls a contract that can receive a callback during the migrator's liquidity-provision call (e.g., a UniswapV3MintCallback or a token `transfer` hook if the quote token is ERC-20)
- The ordering inside `PORTAL_TRADE_V2` places the external migrator call BEFORE `_setTokenStatus(DEX)` is written to storage (this is the unverified precondition)

**attacker-controlled inputs:**
- `ExactInputParams` with `inputToken` = quote token, `outputToken` = target token, `amountIn` = exactly the remaining amount to hit `dexSupplyThresh`
- A malicious migrator callback or ERC-20 `transfer` hook that re-enters `swapExactInput`

**call sequence:**
1. Attacker calls `swapExactInput` → `_getSwapImplementation` reads `status = Tradable` → `_delegateToImpl(PORTAL_TRADE_V2)`
2. Inside `PORTAL_TRADE_V2`: buy is processed, `circulatingSupply` reaches `dexSupplyThresh`, migration is triggered
3. `PORTAL_TRADE_V2` calls external migrator (status still `Tradable` in storage if `_setTokenStatus(DEX)` has not yet been written)
4. Migrator callback re-enters `swapExactInput`
5. `_getSwapImplementation` reads `status = Tradable` (unchanged) → routes to `PORTAL_TRADE_V2` again
6. Reentrant buy executes at bonding-curve price; attacker receives tokens at pre-migration price
7. Outer call completes, `_setTokenStatus(DEX)` is written, liquidity is added to DEX pool

## Why Existing Checks Fail
The only status-based guard is inside `_getSwapImplementation`:

```solidity
// Portal.sol lines 64-75
if (state.status == TokenStatus.DEX) {
    return PORTAL_DEX_ROUTER;
} else if (state.status == TokenStatus.Tradable) {
    return PORTAL_TRADE_V2;
}
```

This is a routing check, not a reentrancy lock. It only blocks the reentrant buy if `_setTokenStatus(DEX)` has already been committed to storage before the external migrator call. If the ordering is reversed (external call first, status write second), the check reads stale `Tradable` state and routes the reentrant call to `PORTAL_TRADE_V2` at bonding-curve price.

There is no `ReentrancyGuardUpgradeable` or `nonReentrant` modifier anywhere in `Portal.sol` or `PortalBase.sol`. [1](#0-0) [2](#0-1) 

The `_setTokenStatus` function is a plain storage write with no lock semantics. [3](#0-2) 

## Rejection Checks
**expected behavior checked:** Buying at bonding-curve price after migration is triggered is not documented as expected behavior; the invariant is that no bonding-curve trades occur after the DEX threshold is crossed.

**prior report checked:** Not determinable from available source; RESEARCHER.md was not read (out of tool calls).

**README/NatSpec checked:** No NatSpec on `swapExactInput` in Portal.sol documents a reentrancy window as acceptable. [4](#0-3) 

**unsupported assumption checked:** The critical unverified assumption is the ordering of `_setTokenStatus(DEX)` vs the external migrator call inside `PORTAL_TRADE_V2`. That implementation is not present in this repository (`PORTAL_TRADE_V2` is an immutable address set at construction time and its source is not indexed here). If `_setTokenStatus(DEX)` is written before the external call, the attack collapses because the reentrant call is routed to `PORTAL_DEX_ROUTER` and fails on an unfunded pool.
