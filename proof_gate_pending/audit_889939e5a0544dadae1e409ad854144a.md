The sole guard at the Portal entry point is `onlyIfBitFlagsSet(CB_BIT_MASK_GLOBAL_SWITCH)`, which checks only that the protocol is not globally halted. No role check (`DEFAULT_ADMIN_ROLE`, `ROLLER_ROLE`, or any other) is applied before the delegatecall to `PORTAL_ROLLER`. [1](#0-0) 

The NatSpec in `IPortal.sol` documents the intended restriction: *"Only the roller or default admin can call this function."* [2](#0-1) 

However, no `ROLLER_ROLE` constant is defined anywhere in the indexed source tree — `grep` across all files returns zero matches for `ROLLER_ROLE`. The only roles defined in `PortalBase` are `GUARDIAN_ROLE`, `TAX_MANAGER_ROLE`, `TAX_GUARDIAN_ROLE`, `TOKEN_FLAP_FEE_SETTER_ROLE`, and `MODERATOR_ROLE`. [3](#0-2) 

The `PORTAL_ROLLER` implementation contract source is **not present** in this repository (the verified source files are `Portal.sol`, `PortalBase.sol`, `PortalCommon.sol`, and the listed interfaces/libraries). Whether the roller implementation itself enforces a caller check on `delegateClaim` cannot be determined from the indexed code alone.

## Attacker Path
**preconditions:**
- Attacker is the `_tokenBeneficiaries[token]` for some launched token (e.g., they created the token or `setTokenBeneficiary` was called to point to them), OR the roller implementation sends fees to the beneficiary regardless of who calls.
- The roller implementation does **not** enforce a role check on `delegateClaim` (unverified — this is the key unknown).
- Global switch bit is set (protocol not halted).

**attacker-controlled inputs:**
- `token` address (any launched token where attacker is beneficiary)

**call sequence:**
1. Attacker EOA calls `Portal.delegateClaim(token)` directly.
2. Portal passes `onlyIfBitFlagsSet` check.
3. `_delegateToImpl(PORTAL_ROLLER)` executes the roller's `delegateClaim` logic in Portal's storage context.
4. If no role check in roller: LP fees are collected and sent to `_tokenBeneficiaries[token]` (attacker-controlled address).

## Why Existing Checks Fail
The Portal-level entry point applies only the global-switch guard. The role enforcement documented in NatSpec is supposed to live inside the `PORTAL_ROLLER` delegatecall target. If the roller omits or misconfigures that check, the Portal provides no fallback guard. There is no `ROLLER_ROLE` defined in the codebase, which raises the question of what role identifier the roller would even check against. [4](#0-3)
