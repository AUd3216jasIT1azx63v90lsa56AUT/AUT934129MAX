## Attacker Path

**preconditions:**
- A token is in `TokenStatus.Tradable` with `circulatingSupply` just below `dexSupplyThresh`.
- The attacker controls a contract that will receive a V3/PCS-Infinity mint callback during liquidity provision.

**attacker-controlled inputs:**
- A `swapExactInput` call sized to push `circulatingSupply >= dexSupplyThresh`, triggering migration inside `PORTAL_TRADE_V2`.

**call sequence:**
1. Attacker calls `swapExactInput` (buy) with exact amount to cross `dexSupplyThresh`.
2. Inside `PORTAL_TRADE_V2` (delegatecall context), migration is triggered.
3. If `PORTAL_TRADE_V2` calls the external migrator (V3 pool `mint`) **before** calling `_setTokenStatus(DEX)`, the pool issues a `uniswapV3MintCallback` / `pancakeV3MintCallback` to Portal.
4. Portal's `receive()` or callback handler re-enters `swapExactInput`.
5. `_getSwapImplementation` reads storage: status is still `Tradable` â†’ routes to `PORTAL_TRADE_V2`.
6. Re-entrant buy executes at bonding-curve price against a reserve that has already been (partially) committed to the migrator, acquiring tokens at a stale/underpriced rate.

## Why Existing Checks Fail

**No reentrancy guard anywhere in Portal source.** The grep for `nonReentrant`, `ReentrancyGuard`, `_status`, `_locked` returned zero matches in Portal's own contracts â€” only OpenZeppelin library internals. [1](#0-0) 

**Routing check is pre-delegatecall and reads live storage.** `_getSwapImplementation` reads `_packedTokenStates` at call time. If `_setTokenStatus(DEX)` has not yet been written when the re-entrant call arrives, the routing sends the re-entrant call to `PORTAL_TRADE_V2` (bonding curve), not `PORTAL_DEX_ROUTER`. [5](#0-4) 

**`_setTokenStatus` is a plain storage write with no lock.** It provides no mutual-exclusion guarantee against concurrent (re-entrant) execution. [3](#0-2) 

**Portal explicitly implements V3 mint callbacks**, meaning the V3 pool can call back into Portal during `mint()`, which is the standard liquidity-provision step in a migration. [4](#0-3) 

## Rejection Checks

**expected behavior checked:** Bonding-curve buys after migration is triggered are not expected protocol behavior; the invariant is that no bonding-curve trades occur once migration begins.

**prior report checked:** Not found in available source/NatSpec.

**README/NatSpec checked:** No NatSpec on `swapExactInput` or migration path documents this as intentional or mitigated.

**unsupported assumption checked:** The critical unknown is the **ordering of `_setTokenStatus(DEX)` vs the external migrator `mint()` call** inside `PORTAL_TRADE_V2`. `PORTAL_TRADE_V2` is an immutable facet address set at construction time and its source is **not present in this repository** â€” only the interface ABI is available. If `PORTAL_TRADE_V2` calls `_setTokenStatus(DEX)` **before** the external `mint()`, the re-entrant `swapExactInput` would be routed to `PORTAL_DEX_ROUTER` (unfunded pool â†’ revert), and the attack fails. The scenario is only exploitable if the status write comes **after** the external call. This ordering cannot be confirmed without the `PORTAL_TRADE_V2` source.

    function _getSwapImplementation(address inputToken, address outputToken)
        internal
        view
        returns (address implementation)
    {
        // Determine if this is a buy or sell
        bool isBuy = _quoteTokenConfigurations[inputToken].enabled == 1;
        address baseToken = isBuy ? outputToken : inputToken;

        // Get token state to check status
        PackedTokenStateV2 memory state = _getTokenState(baseToken);

        if (state.status == TokenStatus.DEX) {
            // Token is listed on DEX, use PortalDexRouter
            return PORTAL_DEX_ROUTER;
        } else if (state.status == TokenStatus.Tradable) {
            // Token is still in bonding curve, use PortalTradeV2
            return PORTAL_TRADE_V2;
        } else {
            // Invalid or other status, will fail in implementation
            return PORTAL_TRADE_V2; // Let PortalTradeV2 handle the error
        }
    }
