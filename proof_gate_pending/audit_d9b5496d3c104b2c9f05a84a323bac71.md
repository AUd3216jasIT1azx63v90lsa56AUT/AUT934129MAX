The symmetric buy-side function `estimateSupplyV2` only rounds once (a single `divWadUp`), while `estimateReserveV2` rounds twice. [2](#0-1) 

## Attacker Path

**preconditions:**
- A token is launched with a non-18-decimal ERC20 quote token (e.g. USDC, 6 decimals; `scaleFactor = 10^12`).
- The token is still on the bonding curve (`TokenStatus.Tradable`), routing `swapExactInput` to `PORTAL_TRADE_V2`.

**attacker-controlled inputs:**
- `inputToken = quoteTokenAddress` (e.g. USDC), `outputToken = bondingCurveToken`, `inputAmount = N` (buy)
- `inputToken = bondingCurveToken`, `outputToken = quoteTokenAddress` (sell)

**call sequence:**
1. Attacker calls `swapExactInput` (buy): pays `inputAmount` USDC → receives tokens.
   - PORTAL_TRADE_V2 computes `newSupply = estimateSupplyV2(curve, state.reserve + inputAmount, 6)` (one round-up).
   - Sets `state.reserve = estimateReserveV2(curve, newSupply, 6)` (two round-ups).
   - `state.reserve` can be `state.reserve_before + inputAmount + 1` while `quoteBalance` only increases by `inputAmount`.
2. Attacker calls `swapExactInput` (sell): returns tokens → receives USDC.
   - Payout = `state.reserve - estimateReserveV2(curve, newSupply_after_sell, 6)`.
   - If `state.reserve` is inflated by 1 unit, payout is inflated by 1 unit.
   - Attacker receives `inputAmount + 1` USDC having deposited `inputAmount`.

## Why Existing Checks Fail

The NatSpec on `estimateReserve` (18-decimal) states the intended invariant: *"currReserve >= estimateReserve_without_roundup(currSupply)"* — a single `divWadUp` is sufficient to maintain this. [3](#0-2) 

For `estimateReserveV2` with `reserveDecimals < 18`, a **second** round-up is applied at line 156. The 18-decimal `scaledReserve` can be up to `scaleFactor - 1 = 10^12 - 1` wei above the true value from `divWadUp` alone; the second ceiling division then adds another full unit in native-decimal space. The result is that `estimateReserveV2(curve, supply, 6) * scaleFactor` can exceed the true 18-decimal reserve by up to `2 * (scaleFactor - 1)`, i.e. up to **2 units** of the 6-decimal token per call, rather than the intended ≤1 unit.

No guard in the available source (`Portal.sol`, `PortalBase.sol`, `PortalCommon.sol`) checks that `state.reserve <= quoteTokenBalance` after each trade. The `_setTokenReserve` setter performs no such validation. [4](#0-3) 

**Critical gap:** `estimateReserveV2` and `estimateSupplyV2` are defined only in `Curve.sol` and are not called from any `src/` Solidity file in this repository. The actual call sites live in the `PORTAL_TRADE_V2` implementation contract, which is not present in the source tree. The exact accounting (whether `state.reserve` is set to `estimateReserveV2(curve, newSupply, 6)` or to `old_reserve + inputAmount`) cannot be confirmed without that source.

## Rejection Checks

**expected behavior checked:** The NatSpec explicitly says rounding up is "more favorable to the protocol." The double round-up is not documented as intentional; the 18-decimal path only rounds once. This is a design asymmetry, not documented expected behavior.

**prior report checked:** No prior report found in the indexed source.

**README/NatSpec checked:** No NatSpec on `estimateReserveV2` acknowledges the second rounding step as intentional or discusses its interaction with `state.reserve` accounting.

**unsupported assumption checked:** The claim depends on PORTAL_TRADE_V2 setting `state.reserve = estimateReserveV2(curve, newSupply, decimals)` after each trade (curve-derived, not deposit-tracked). This is the standard pattern for bonding-curve reserve accounting in this codebase (consistent with how `estimateReserve` is described), but cannot be confirmed without the implementation source. If PORTAL_TRADE_V2 instead tracks `state.reserve += inputAmount` on buys, the invariant violation direction reverses (sell payout could underflow/revert rather than over-pay).

```text
    /// @notice estimate the reserve given the supply
    /// @dev This function returns a roundup value, because we want the following invariant to hold:
    ///         currReserve >= estimateReserve_without_roudup(currSupply)
    ///
    ///      This function is used when a user wants to sell tokens, a rounded up value
    ///      is more favorable to the protocol.
    function estimateReserve(Curve memory curve, uint256 supply) internal pure returns (uint256 reserve) {
        if (supply > TOTAL_SUPPLY) {
            revert SupplyExceedsTotalSupply(supply);
        }

        // eth = k/(h + 1e9 - s) - r
        // Round up for protocol safety when selling
        reserve = FixedPointMathLib.divWadUp(curve.k, TOTAL_SUPPLY + curve.h - supply) - curve.r;
    }
```
