The `else` branch is reached for any `TokenStatus` that is neither `DEX` nor `Tradable` — including `Staged`, `Killed`, `InDuel`, and `Invalid`. For a `Staged` token, `_getSwapImplementation` returns `PORTAL_TRADE_V2` without reverting. [1](#0-0) 

`swapExactInput` then unconditionally delegates to that address: [2](#0-1) 

## Attacker Path
**preconditions:**
- `SALE_FORGE` has called `stageNewTokenV5`, setting the token's `PackedTokenStateV2.status = TokenStatus.Staged` and `reserve = 0`, `circulatingSupply = 0`. `commitNewTokenV5` has not yet been called.
- For V2+ token versions (`TOKEN_V2_PERMIT`, `TOKEN_TAXED`, etc.), the full token supply is pre-minted to Portal at staging time, so Portal holds the token balance.
- The global `bitFlags` has `CB_BIT_MASK_GLOBAL_SWITCH` set (normal operating state).

**attacker-controlled inputs:**
- `ExactInputParams { inputToken: address(0), outputToken: <staged_token>, inputAmount: X, minOutputAmount: 0, permitData: "" }`

**call sequence:**
1. Attacker calls `swapExactInput(params)` with `msg.value > 0`.
2. `_getSwapImplementation(address(0), <staged_token>)` reads `state.status == TokenStatus.Staged`, falls into the `else` branch, returns `PORTAL_TRADE_V2`.
3. `_delegateToImpl(PORTAL_TRADE_V2)` executes `PORTAL_TRADE_V2`'s buy logic in Portal's storage context with `reserve = 0` and `circulatingSupply = 0`.
4. If `PORTAL_TRADE_V2` does not check `state.status != TokenStatus.Tradable` before proceeding, the bonding curve is evaluated at `reserve = 0`, yielding the maximum token output for the given BNB input (initial curve price, most favorable to buyer).
5. Portal transfers tokens from its own balance to the attacker; `reserve` and `circulatingSupply` are updated as if a normal buy occurred.

## Why Existing Checks Fail
`_getSwapImplementation` performs no status guard for `Staged` — the comment "will fail in implementation" defers the check entirely to `PORTAL_TRADE_V2`. [3](#0-2) 

The `TokenNotTradable` error is declared in the interface: [4](#0-3) 

However, **the `PORTAL_TRADE_V2` implementation contract is not present in this repository's source files** (`src/` contains only `Portal.sol`, `PortalBase.sol`, `PortalCommon.sol`, and the interfaces/libraries). Whether `PORTAL_TRADE_V2` actually checks `state.status == TokenStatus.Tradable` before executing the bonding-curve buy — and reverts with `TokenNotTradable` for `Staged` — cannot be confirmed from the available indexed source. The developer's comment expresses intent but is not a code-level guarantee.

`stageNewTokenV5` is restricted to `SALE_FORGE`: [5](#0-4) 

This means staged tokens only exist when `SALE_FORGE` has staged one but not yet committed it — a real but narrow time window on the live BSC deployment.

## Rejection Checks
**expected behavior checked:** The comment "will fail in implementation" is the developer's stated intent, not a code-level guard. If `PORTAL_TRADE_V2` omits the status check, the behavior is not expected.

**prior report checked:** No prior report found in the indexed source.

**README/NatSpec checked:** `TokenStatus.Staged` is documented as "address determined, but not yet created" — trading it violates the stated invariant. [6](#0-5) 

**unsupported assumption checked:** The assumption that `PORTAL_TRADE_V2` lacks the status check is unverified — it is the core unknown requiring local proof.

## Local Proof Required
**test type:** Foundry fork test against the live BSC proxy (`0xe2ce6ab80874fa9fa2aae65d277dd6b8e65c9de0`) or a local unit test with a mock `PORTAL_TRADE_V2`.

**test file to add:** `test/StagedTokenSwap.t.sol`

**test setup:**
1. Deploy Portal with a mock `PORTAL_TRADE_V2` that does NOT check `state.status` (to isolate the router-level guard question), or fork BSC and use the live `PORTAL_TRADE_V2`.
2. Impersonate `SALE_FORGE` and call `stageNewTokenV5` to create a token in `Staged` status with `reserve = 0`.
3. As an unprivileged attacker, call `swapExactInput({ inputToken: address(0), outputToken: stagedToken, inputAmount: 1 ether, minOutputAmount: 0 })` with `msg.value = 1 ether`.

**expected assertion (safe behavior):** Transaction reverts with `TokenNotTradable(stagedToken)`.

**failure condition (vulnerability confirmed):** Transaction succeeds; attacker receives tokens at the initial bonding-curve price with `reserve = 0`, extracting tokens from Portal's balance without the token having been properly initialized. Any subsequent `commitNewTokenV5` would then start from a corrupted reserve/supply state, enabling further over-extraction.
