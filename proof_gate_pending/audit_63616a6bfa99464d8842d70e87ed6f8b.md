## Attacker Path
**preconditions:**
- Attacker controls an address that is `_tokenBeneficiaries[token]` for some token with a non-zero `lpLocks[token]` entry that has accumulated uncollected fees
- PORTAL_ROLLER's `claim` implementation sends ETH to the beneficiary *before* zeroing/updating the relevant accounting state (CEI violation — **unconfirmed without roller source**)
- No reentrancy guard exists anywhere in the Portal proxy or its storage context

**attacker-controlled inputs:**
- `token` address passed to `claim(token)`
- A receiver contract deployed as the beneficiary whose `receive()` re-calls `Portal.claim(token)`

**call sequence:**
1. Attacker sets their malicious receiver contract as `_tokenBeneficiaries[token]` (via `setTokenBeneficiary`)
2. Attacker calls `Portal.claim(token)`
3. Portal delegatecalls into PORTAL_ROLLER (runs in Portal's storage context)
4. PORTAL_ROLLER calls the locker's `collectFees`, which sends ETH to Portal
5. PORTAL_ROLLER then sends ETH to the beneficiary (attacker contract) — **before** clearing the claimable state
6. Attacker's `receive()` re-enters `Portal.claim(token)`
7. PORTAL_ROLLER sees the same uncleaned state and pays out again
8. Loop repeats until gas exhaustion or balance depletion

## Why Existing Checks Fail
A full grep of all `src/**/*.sol` files returns **zero matches** for `nonReentrant`, `ReentrancyGuard`, `_reentrancyGuard`, `_status`, or `_locked`. [1](#0-0)  There is no reentrancy guard on `claim()`, on `_delegateToImpl`, or on `receive()`. The `onlyIfBitFlagsSet` modifier only checks a feature-enable flag — it provides no reentrancy protection. [6](#0-5) 

**Critical gap:** The PORTAL_ROLLER implementation is **not present in this repository**. The repo contains only `src/Portal.sol`, `src/PortalBase.sol`, `src/PortalCommon.sol`, and interface/library files. PORTAL_ROLLER is an immutable address set at construction time pointing to an external contract. [7](#0-6)  Without the roller source, it is impossible to confirm whether:
- The roller clears `lpLocks` / beneficiary-claimable state before or after the ETH transfer
- The roller has its own internal reentrancy guard
- The locker's `collectFees` callback path actually routes ETH through Portal's `receive()` or directly to the beneficiary

## Rejection Checks
**expected behavior checked:** Plain `receive() external payable {}` is expected for a contract that holds BNB; the absence of a reentrancy guard on `claim` is not documented as intentional. [2](#0-1) 

**prior report checked:** No prior report or known-issues file is present in the repository.

**README/NatSpec checked:** No NatSpec on `claim` or `receive` documents reentrancy safety or CEI compliance. [1](#0-0) 

**unsupported assumption checked:** The entire attack depends on the roller having a CEI violation. This is the blocking unknown — the roller source is not indexed and cannot be confirmed from available code.

## Local Proof Required
**test type:** Foundry fork test on BSC

**test file to add:** `test/ClaimReentrancy.t.sol`

**test setup:**
1. Fork BSC at block 108382650 against proxy `0xe2ce6ab80874fa9fa2aae65d277dd6b8e65c9de0`
2. Identify a live token where `lpLocks[token].locks[0]` or `locks[1]` is non-zero and has accumulated fees (query `getLocks(token)` on the proxy)
3. Obtain or impersonate the current `_tokenBeneficiaries[token]` address, or call `setTokenBeneficiary` to set it to the attacker contract (check if `setTokenBeneficiary` is permissioned — if it requires the current beneficiary or token creator, set up accordingly)
4. Deploy a malicious receiver contract whose `receive()` calls `Portal.claim(token)` up to N times, tracking ETH received per call
5. Record `address(portal).balance` before the first `claim` call

**expected assertion:**
```
assertEq(address(portal).balance_before - address(portal).balance_after, legitimateFeeAmount);
assertEq(attackerContract.totalReceived(), legitimateFeeAmount); // fails if reentrancy drains more
```

**failure condition:** If `attackerContract.totalReceived() > legitimateFeeAmount` and `portal.balance` drops by more than one fee cycle, the reentrancy is confirmed. If the roller clears state before the ETH transfer (or has its own guard), all re-entrant calls will return `(0, 0)` and the test passes cleanly — confirming REJECT.
