## Exact Code Path

symbols/lines: lines 54–76

```solidity
} else {

```

The `else` branch is reached for any token whose status is neither `TokenStatus.DEX` nor `TokenStatus.Tradable` — including `TokenStatus.Staged` and `TokenStatus.Invalid`. The Portal itself performs no status guard; it delegates the rejection responsibility entirely to `PORTAL_TRADE_V2`. [1](#0-0) 

## Attacker Path
**preconditions:**
- `SALE_FORGE` has called `stageNewTokenV5`, setting the token's status to `TokenStatus.Staged` in `_packedTokenStates`. The token address is predetermined (CREATE2), but `commitNewTokenV5` has not yet been called, so `reserve == 0` and `circulatingSupply == 0`.
- The attacker does not need to call `stageNewTokenV5` themselves — they only need to know the staged token address, which is deterministic and observable on-chain.

**attacker-controlled inputs:**
- `ExactInputParams.inputToken` = a valid quote token (e.g., native BNB path)
- `ExactInputParams.outputToken` = the staged token address
- `msg.value` = some BNB

**call sequence:**
1. SALE_FORGE calls `stageNewTokenV5(...)` → token status set to `Staged`, reserve = 0, supply = 0.
2. Attacker calls `swapExactInput({inputToken: quoteToken, outputToken: stagedToken, ...})`.
3. `_getSwapImplementation` reads `state.status == TokenStatus.Staged`, which is neither `DEX` nor `Tradable`, falls into the `else` branch, and returns `PORTAL_TRADE_V2`.
4. `_delegateToImpl(PORTAL_TRADE_V2)` is called.
5. Whether the call succeeds or reverts depends entirely on whether `PORTAL_TRADE_V2` checks `state.status == TokenStatus.Tradable` before executing bonding-curve logic. [2](#0-1) [3](#0-2) 

## Why Existing Checks Fail
`_getSwapImplementation` has no explicit guard for `TokenStatus.Staged` or `TokenStatus.Invalid`. The only two affirmative branches are `DEX` and `Tradable`; everything else silently falls through to `PORTAL_TRADE_V2` with a comment that the implementation will "handle the error." [1](#0-0) 

`stageNewTokenV5` and `commitNewTokenV5` are correctly restricted to `SALE_FORGE`, so an attacker cannot manufacture a Staged token. However, once SALE_FORGE stages a token, the window between `stageNewTokenV5` and `commitNewTokenV5` is exploitable if `PORTAL_TRADE_V2` does not independently check the status. [4](#0-3) 

The `PORTAL_TRADE_V2` source is not present in this repository (it is a separate delegatecall facet). The comment "Let PortalTradeV2 handle the error" is the only evidence of the intended guard. If `PORTAL_TRADE_V2` checks `state.status == TokenStatus.Tradable` before proceeding, the call reverts safely. If it does not — or if it only checks that the token is not `Invalid` — it would attempt bonding-curve arithmetic on a zero-reserve, zero-supply token, potentially minting tokens for free or causing an accounting corruption that benefits the attacker.

## Rejection Checks
**expected behavior checked:** The comment explicitly acknowledges the else-branch is an error path and delegates handling to `PORTAL_TRADE_V2`. Whether that delegation is sufficient is the open question.

**prior report checked:** No prior report found in this repository.

**README/NatSpec checked:** No NatSpec on `_getSwapImplementation` documents the Staged-status behavior beyond the inline comment.

**unsupported assumption checked:** The scenario does not require the attacker to call `stageNewTokenV5` (restricted to `SALE_FORGE`). It only requires a Staged token to exist, which is a normal protocol state during the two-step launch flow.

## Local Proof Required
**test type:** Foundry fork test (BSC mainnet fork at block ≥ 108382650) or unit test with mocked state.

**test file to add:** `test/SwapStagedToken.t.sol`

**test setup:**
1. Deploy or fork the Portal proxy at `0xe2ce6ab80874fa9fa2aae65d277dd6b8e65c9de0`.
2. Impersonate `SALE_FORGE` and call `stageNewTokenV5(...)` to produce a token in `TokenStatus.Staged` with `reserve == 0` and `circulatingSupply == 0`.
3. As an unprivileged EOA, call `swapExactInput({inputToken: address(0) or WBNB, outputToken: stagedToken, amountIn: 1 ether, minAmountOut: 0, recipient: attacker})` with `msg.value = 1 ether`.

**expected assertion:**
- **Safe:** call reverts with a `TokenNotTradable`-style error from `PORTAL_TRADE_V2`.
- **Vulnerable:** call succeeds; attacker receives tokens; Portal's BNB balance increases but token reserve/supply accounting is corrupted, or attacker receives tokens at zero effective cost.

**failure condition:** If the call does not revert and the attacker's token balance increases, the invariant that Staged tokens must not be tradeable is broken and the finding is confirmed as a fund-extraction vulnerability.
