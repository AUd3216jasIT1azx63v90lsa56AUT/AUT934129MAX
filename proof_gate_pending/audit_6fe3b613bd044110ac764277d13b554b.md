## Attacker Path
**preconditions:**
- A token exists whose `quoteTokenAddress` is USDT and `nativeToQuoteSwapEnabled = true`
- The BNB→USDT pool routed through `MULTI_DEX_ROUTER` has enough depth to accept the swap but returns fewer USDT than the BNB-equivalent at spot price (normal slippage)

**attacker-controlled inputs:**
- `params.inputToken = address(0)` (native BNB)
- `params.outputToken = <USDT-quote token>`
- `params.inputAmount = msg.value` (any non-trivial BNB amount)
- `params.minOutputAmount` set low enough to not revert

**call sequence:**
1. Attacker calls `swapExactInput{value: X BNB}(...)` on the Portal proxy
2. Portal delegates to the swap implementation
3. Implementation calls `MULTI_DEX_ROUTER` to swap X BNB → Y USDT (Y < X * spot_price due to slippage)
4. **Claimed bug:** implementation credits `reserve += f(msg.value)` using the BNB amount rather than `reserve += Y` (actual USDT received)
5. `state.reserve` is now overstated relative to Portal's actual USDT balance
6. Attacker (or any seller) calls `swapExactInput` to sell the token back; the bonding curve pays out USDT based on the inflated reserve, extracting more USDT than was actually deposited

## Why Existing Checks Fail
The `minOutputAmount` slippage guard on `swapExactInput` protects the *caller's token output*, not the reserve accounting integrity. If the implementation uses `msg.value` (or a BNB-denominated conversion) to update `state.reserve` instead of the actual ERC20 amount returned by `MULTI_DEX_ROUTER`, no on-chain check prevents the reserve from being overstated. The `_setTokenReserve` helper in `PortalBase.sol` simply writes whatever value it is given. [4](#0-3) 

## Rejection Checks
**expected behavior checked:** Slippage on the internal BNB→USDT swap is not documented as an accepted reserve discrepancy; the invariant that `state.reserve` equals actual quote-token balance is stated in the interface NatSpec.

**prior report checked:** Not found in the indexed source.

**README/NatSpec checked:** `IPortal.sol` NatSpec for `TokenStateV3.reserve` says "The reserve amount of the quote token held by the bonding curve" — this implies it must equal actual USDT held, not a BNB-equivalent estimate.

**unsupported assumption checked:** The critical assumption — that the implementation credits `msg.value` rather than actual USDT received — **cannot be verified from the indexed source files** because the buy-path implementation is in a delegated contract not present in the repo index. This is the reason for NEEDS_LOCAL_PROOF rather than HIGH_CONFIDENCE_CANDIDATE.

## Local Proof Required
**test type:** Foundry fork test on BSC mainnet

**test file to add:** `test/NativeToQuoteSwapReserveAccounting.t.sol`

**test setup:**
1. Fork BSC at a recent block
2. Identify a live USDT-quote token with `nativeToQuoteSwapEnabled = true` (query `getTokenV3` or `getTokenV8` on the proxy)
3. Record `state.reserve` and Portal's USDT balance before the swap
4. Call `swapExactInput{value: 1 ether}(inputToken=address(0), outputToken=<token>, minOutputAmount=0)`
5. Record `state.reserve` and Portal's USDT balance after the swap

**expected assertion (no bug):**
```
assertEq(
    state_after.reserve - state_before.reserve,
    portal_usdt_balance_after - portal_usdt_balance_before
);
```

**failure condition:** If `state_after.reserve - state_before.reserve` exceeds `portal_usdt_balance_after - portal_usdt_balance_before` by any non-trivial amount, the reserve is overstated and the bug is confirmed. A subsequent sell call should then be able to extract more USDT than was deposited, confirming fund extraction.
