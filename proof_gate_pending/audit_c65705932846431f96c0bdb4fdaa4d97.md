## Attacker Path
**preconditions:**
- Attacker is the current `_tokenBeneficiaries[token]` for a token whose `lpLocks[token]` is non-zero and has accumulated uncollected LP fees
- Attacker controls a contract with a malicious `receive()` that re-enters `Portal.claim(token)`

**attacker-controlled inputs:**
- `token` address passed to `claim()`
- Attacker's contract address registered as beneficiary (either at token creation or via `setTokenBeneficiary` if attacker is the current beneficiary)

**call sequence:**
1. Attacker calls `Portal.claim(token)` from malicious contract
2. Portal dispatches via `_delegateToImpl(PORTAL_ROLLER)` — roller logic runs in Portal's storage context
3. Roller checks `_tokenBeneficiaries[token] == msg.sender` ✓
4. Roller calls locker's `collectFees()` — locker sends ETH to Portal
5. Portal's `receive() external payable {}` is triggered
6. Attacker's `receive()` re-enters `Portal.claim(token)` before roller clears `_tokenBeneficiaries` or marks fees as claimed
7. If roller has not yet updated state, the beneficiary check passes again and fees are re-collected

## Why Existing Checks Fail
**No reentrancy guard anywhere in `src/`:** A full grep for `nonReentrant`, `ReentrancyGuard`, `_status`, `_locked`, `reentrancy`, and `mutex` across all `src/**/*.sol` files returns zero matches. Neither `Portal.sol`, `PortalBase.sol`, nor `PortalCommon.sol` import or apply any reentrancy protection. [3](#0-2) 

**Plain `receive()`:** Portal's `receive()` is a single-line unconditional payable with no guard, no state check, and no call depth tracking. Any ETH sent to Portal during a `claim` execution (e.g., from a locker's `collectFees`) will silently succeed and return control to the caller. [2](#0-1) 

**Critical gap — PORTAL_ROLLER implementation not in repository:** The `PORTAL_ROLLER` is an external implementation contract passed as an immutable address at construction time. [4](#0-3) [5](#0-4) 

The `src/` directory contains only `Portal.sol`, `PortalBase.sol`, `PortalCommon.sol`, and interfaces — the roller's actual `claim` logic (beneficiary check, fee collection call, state update ordering, ETH transfer) is not available in this repository. Whether the roller applies checks-effects-interactions (clearing `_tokenBeneficiaries` or a "claimed" flag before the locker call) or performs the ETH transfer last cannot be confirmed from the available source. This is the single blocking unknown.

## Rejection Checks
**expected behavior checked:** `claim` paying out LP fees to the beneficiary is expected; paying out more than once per call via reentrancy is not. [6](#0-5) 

**prior report checked:** No prior report or known-issue note found in the available source files. 

**README/NatSpec checked:** IRoller NatSpec says "Only the beneficiary of the token can call this function" — no mention of reentrancy protection or CEI ordering. [7](#0-6) 

**unsupported assumption checked:** The assumption that `locker.collectFees()` routes ETH through `Portal.receive()` is architecturally consistent with the locker's `collectAddress` being set to Portal (per the `IV4Locker` NatSpec), but requires fork-test confirmation. [8](#0-7)

**test file to add:** `test/fork/ClaimReentrancy.t.sol`

**test setup:**

2. Identify a token address where `lpLocks[token]` is non-zero and has accumulated fees (query `getLocks(token)` and check locker balance)
3. Deploy a `MaliciousReceiver` contract whose `receive()` calls `Portal.claim(token)` and counts re-entry depth
4. Use `vm.prank` or a live beneficiary key to set `MaliciousReceiver` as `_tokenBeneficiaries[token]` (or find a token where attacker already controls the beneficiary)
5. Record `Portal.balance` before the first `claim` call

**expected assertion:**
- If vulnerable: `Portal.balance` decreases by more than the single legitimate fee amount; `BeneficiaryClaimed` event emitted more than once in the same transaction
- If protected: second `claim` call reverts (reentrancy guard) or returns `(0, 0)` (CEI pattern cleared state before transfer)

**failure condition:** Test passes (reentrancy succeeds and Portal balance is over-drained) → HIGH_CONFIDENCE_CANDIDATE; test reverts on re-entry → REJECT
