Looking at the `updateTokenPoolInfo` function in Portal.sol and the delegatecall architecture, I need to determine whether role enforcement exists.

**Key finding in Portal.sol:** [1](#0-0) 

The Portal.sol entry point for `updateTokenPoolInfo` carries only `onlyIfBitFlagsSet(CB_BIT_MASK_GLOBAL_SWITCH)` â€” a global halt check, not a role check. The actual role enforcement (if any) lives entirely inside the `PORTAL_DEX_ROUTER` delegatecall target.

**The `onlyIfBitFlagsSet` modifier:** [2](#0-1) 

This only checks whether the global feature bit is set â€” it is not an access-control guard.

**`_dexPools` storage:** [3](#0-2) 

**`DEX_ROUTER_MANAGER_ROLE` is defined only in the interface:** [4](#0-3) 

The `PORTAL_DEX_ROUTER` immutable is set at construction time: [5](#0-4) 

**Critical gap:** The `PORTAL_DEX_ROUTER` implementation source (`PortalDexRouter.sol` or equivalent) is **not present in the indexed source files** (`src/Portal.sol`, `src/PortalBase.sol`, `src/PortalCommon.sol`, interfaces, libraries). The role enforcement â€” if it exists â€” lives entirely in that unindexed delegatecall target. Without reading it, the presence or absence of a `hasRole(DEX_ROUTER_MANAGER_ROLE, msg.sender)` check cannot be confirmed from source alone.

**Swap routing confirms `_dexPools` is used for DEX-status tokens:** [6](#0-5) 

When a token's status is `DEX`, `swapExactInput` delegates to `PORTAL_DEX_ROUTER`, which reads `_dexPools[token]` to route the swap. If an attacker could overwrite that mapping with a fake pool, swap proceeds would be misdirected.

## Attacker Path
preconditions: Portal is not halted (global bit set); at least one token has `TokenStatus.DEX`; attacker is an unprivileged EOA with no `DEX_ROUTER_MANAGER_ROLE`

attacker-controlled inputs: `token` = address of a live DEX-graduated token; `poolInfo.pool` = attacker-controlled fake pool contract

call sequence:
1. Attacker calls `Portal.updateTokenPoolInfo(token, fakePoolInfo)` â€” passes `onlyIfBitFlagsSet` check
2. Execution delegates to `PORTAL_DEX_ROUTER` via `_delegateToImpl`
3. If `PORTAL_DEX_ROUTER` lacks a `hasRole(DEX_ROUTER_MANAGER_ROLE, msg.sender)` check, it writes `_dexPools[token] = fakePoolInfo`
4. Subsequent `swapExactInput` calls on that token route to the fake pool, which can drain quote tokens sent by users

## Why Existing Checks Fail
Portal.sol's `updateTokenPoolInfo` applies only `onlyIfBitFlagsSet(CB_BIT_MASK_GLOBAL_SWITCH)`, which is a liveness guard, not an access-control guard. The `DEX_ROUTER_MANAGER_ROLE` is declared in the interface but its enforcement point is exclusively inside the `PORTAL_DEX_ROUTER` delegatecall target â€” a contract whose source is not in the indexed files. If that target omits or misplaces the `hasRole` check, the Portal.sol layer provides no fallback protection.
