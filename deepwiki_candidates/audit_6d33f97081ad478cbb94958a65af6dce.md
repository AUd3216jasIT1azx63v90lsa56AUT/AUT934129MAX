## Attacker Path

**preconditions:**
- Attacker creates a token via any `newToken*` path and sets themselves as `beneficiary` (or is already the beneficiary of an existing token).
- The token has been launched to DEX and has accrued LP fees in the locker.
- The `PORTAL_ROLLER` implementation does not enforce `ROLLER_ROLE` or `DEFAULT_ADMIN_ROLE` on `msg.sender` inside `delegateClaim`.

**attacker-controlled inputs:**
- `token` = address of a token whose `_tokenBeneficiaries[token]` is attacker-controlled.

**call sequence:**
1. Attacker EOA calls `Portal.delegateClaim(token)` directly.
2. Proxy passes `onlyIfBitFlagsSet(CB_BIT_MASK_GLOBAL_SWITCH)` (global switch is on).
3. `_delegateToImpl(PORTAL_ROLLER)` executes the roller's `delegateClaim` logic in Portal's storage context.
4. If no role check exists in the roller, the full claim flow runs: LP fees are collected and sent to `_tokenBeneficiaries[token]` (attacker).
5. Attacker can repeat at will, triggering premature or repeated fee distributions.

## Why Existing Checks Fail

The proxy-level `delegateClaim` applies only `onlyIfBitFlagsSet(CB_BIT_MASK_GLOBAL_SWITCH)` ŌĆö no role check. [3](#0-2) 

`ROLLER_ROLE` does not appear anywhere in the indexed source (`grep` across the entire repo returns zero matches). There is no `bytes32 public constant ROLLER_ROLE` defined in `PortalBase`, `PortalCommon`, or `Portal`. The role check, if it exists at all, must be inside the `PORTAL_ROLLER` implementation contract ŌĆö whose source is **not present in this repository** (`src/` contains only `Portal.sol`, `PortalBase.sol`, `PortalCommon.sol`, and `interfaces/`/`libraries/`).


This means the entire access-control burden for `delegateClaim` falls on the external roller implementation. If that implementation omits the check (or uses a role identifier that is never granted), any EOA can call through.

Note: `claim` is separately restricted to the beneficiary only. `delegateClaim` is architecturally distinct ŌĆö it is intended for rollers/admins to push fees to the beneficiary on a schedule. If it is callable by anyone, the attacker (as beneficiary) can self-trigger at arbitrary frequency, bypassing any roller-controlled timing or batching logic.
