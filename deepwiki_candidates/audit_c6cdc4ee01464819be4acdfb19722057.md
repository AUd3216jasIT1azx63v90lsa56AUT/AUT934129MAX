The Portal-level entry point carries **no access-control modifier** — no `onlyRole(DEFAULT_ADMIN_ROLE)`, no `onlyRole(GUARDIAN_ROLE)`, no caller check of any kind. [1](#0-0) 

The `_tokenBeneficiaries` mapping that would be mutated lives in `PortalBase.sol` at slot 169. [2](#0-1) 

The interface NatSpec documents the intent as admin-only ("Allows the default admin to change the beneficiary of a token"), but NatSpec is not enforcement. [3](#0-2) 

## Attacker Path
**preconditions:**
- Any token that has been created and has a beneficiary set in `_tokenBeneficiaries`
- `PORTAL_ROLLER` implementation does **not** enforce `hasRole(DEFAULT_ADMIN_ROLE, msg.sender)` or `msg.sender == _tokenBeneficiaries[token]` internally (this is the unverified precondition — the `PORTAL_ROLLER` source file is not present in the indexed repository)

**attacker-controlled inputs:**
- `token` = address of any launched token with accumulated LP-fee or beneficiary claims
- `newBeneficiary` = attacker-controlled address

**call sequence:**
1. Attacker calls `Portal.setTokenBeneficiary(token, attackerAddress)` with no special role
2. Portal delegates via `_delegateToImpl(PORTAL_ROLLER)` — no guard fires
3. If `PORTAL_ROLLER` also lacks the check, `_tokenBeneficiaries[token]` is overwritten with `attackerAddress`
4. Attacker calls `Portal.claim(token)` or waits for `delegateClaim` — all future LP-fee and beneficiary payouts route to `attackerAddress`

## Why Existing Checks Fail
The Portal dispatcher applies no role guard before the delegatecall. The only place a guard could exist is inside `PORTAL_ROLLER`. The `PORTAL_ROLLER` source is **not present** in the indexed repository (the source inventory lists only `Portal.sol`, `PortalBase.sol`, `PortalCommon.sol`, interfaces, and libraries — no `PortalRoller.sol`). [1](#0-0) 

This means the access-control assumption in the question — that `PORTAL_ROLLER` lacks the check — cannot be confirmed or refuted from the available source alone. The Portal entry point itself is unguarded, so if `PORTAL_ROLLER` is also unguarded, the full attack path is open.

## Rejection Checks
**expected behavior checked:** NatSpec says "default admin only"; the Portal dispatcher does not enforce this, so it is not expected-behavior-as-implemented. [3](#0-2) 

**prior report checked:** No prior report found in the indexed repository.

**README/NatSpec checked:** NatSpec on the interface states admin-only intent but provides no on-chain enforcement at the Portal level.

**unsupported assumption checked:** The critical assumption — that `PORTAL_ROLLER` itself lacks the guard — cannot be verified without the `PORTAL_ROLLER` source. This is the reason for NEEDS_LOCAL_PROOF rather than HIGH_CONFIDENCE_CANDIDATE.
