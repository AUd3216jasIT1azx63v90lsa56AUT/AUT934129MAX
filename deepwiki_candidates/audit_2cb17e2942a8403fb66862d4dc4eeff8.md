## Attacker Path

**preconditions:**
- Portal is live and `newTokenV6` is enabled (global bit flag set).
- Attacker is an unprivileged token creator with enough BNB to pay the creation fee.
- `MAGIC_DIVIDEND_SELF` (0xfEEDFEEDfeEDFEedFEEdFEEDFeEdfEEdFeEdFEEd) is accepted as a sentinel that resolves to the newly-created tax token's own address post-deployment.

**attacker-controlled inputs:**
- `NewTokenV6Params.dividendToken = 0xfEEDFEEDfeEDFEedFEEdFEEDFeEdfEEdFeEdFEEd`
- `NewTokenV6Params.dividendBps = 5000`
- Token type: `TOKEN_TAXED_V3`

**call sequence:**
1. Attacker calls `portal.newTokenV6(params)` with the above inputs.
2. Portal delegates to `PORTAL_LAUNCHER` â†’ `launcherV6Impl_`.
3. If the `DividendTokenMustEqualQuoteToken` guard is evaluated **before** the sentinel is resolved, it compares `MAGIC_DIVIDEND_SELF != quoteToken` and reverts â€” no bug.
4. If the guard is evaluated **after** sentinel resolution (i.e., `dividendToken` is replaced with the newly-created tax token address before the check), the check compares `taxToken != quoteToken`, which also reverts â€” no bug.
5. The only exploitable path exists if the guard is **skipped entirely** when `dividendToken == MAGIC_DIVIDEND_SELF`, or if the sentinel is resolved to the quote token address by coincidence/design, allowing a self-referential dividend loop.

## Why Existing Checks Fail

The theory is plausible only if the guard fires on the pre-resolution value and the sentinel is treated as a bypass, which is unconfirmed.

**prior report checked:** Not determinable from available source.

**unsupported assumption checked:** The theory assumes the guard does not fire on the sentinel value. This is unverified â€” both orderings (guard before or after resolution) are possible, and one of them (guard before resolution) would cause an immediate revert, making the attack impossible.
